# Hot2 项目性能全面审查与优化建议

> 审查日期：2026-07-18  
> 审查基线：当前工作区源码、Prisma Schema/迁移、Next.js 配置、Nginx/PM2 部署配置及本地 SQLite 运行状态  
> 目标：在不改变现有 Article → Event 业务边界的前提下，优先消除会随数据量和访问量线性恶化的热点。

## 1. 结论摘要

当前项目已完成本轮低复杂度、高体感的主要优化：公开列表和详情已改为有限数据库查询，抓取记录已拆出轻量 Job 轮询，后台页面与设置子页按需拆包并保留已访问页状态，SQLite 已统一初始化为 WAL。高频数据库往返、无上限缓存、隐藏页轮询和路由预取误计浏览等问题也已收敛。

剩余项主要是容量型优化：抓取记录 Source 明细按需加载、Dashboard 完整 SQL 聚合，以及更大规模下的 PushLog 当前状态物化。按当前“数据源 < 20、日文章 < 200、单实例”规模，它们不是当前交互流畅度的阻断项，不应为此引入新状态表、外部缓存或额外服务。

综合判断：

- 当前数据量（本地样本：30 Article、28 Event、数据库约 3 MB）无法暴露规模问题，不能据此判断生产性能安全。
- 公开读取、抓取记录轮询、后台拆包、页面保活和 SQLite 运行参数等 P0/P1 主体改造已实施。
- 不建议现在引入 Redis、消息队列、微服务或更换数据库；后续只在真实数据量和基准指标证明必要时，再实施剩余容量优化。

## 2. 审查范围与方法

### 1.1 当前实施进度（2026-07-18）

已按项目实际规模先完成低复杂度、高体感改造：

- 后台三个顶层页面和设置重型子页面已按访问动态加载；分享海报仅在点击后加载。
- `/admin` 深链直接加载目标页面，抓取记录与情报收件箱之间切换不再整页刷新。
- 已访问的后台顶层页保留组件状态、滚动位置和已加载数据，返回时不重走首屏请求；隐藏的抓取记录和 Dashboard 暂停轮询。
- 收件箱与抓取记录文章详情支持悬停/聚焦预取、并发请求合并和 15 秒短缓存；写操作会失效或刷新缓存。
- 公开详情 metadata 与页面渲染复用同一次详情查询；详情主体按 Event ID 单条读取，关联文章限制 5 条，导航只读取轻字段，不再读取全部正文。
- 公开详情浏览量从 Server Page 渲染移到客户端真实挂载后的延迟上报，避免 Next.js 路由预取、metadata 读取和 API 查询误计浏览；原文点击与浏览量统一为 1 秒内存短聚合。
- 公开列表缓存已限制为最多 50 个 key；新文章 probe 无搜索时使用数据库 `count`。
- 收件箱列表已移除未使用的 facets 和无上限 `all=true` 能力，常规翻页只执行当前页查询与计数。
- 抓取记录按当前日处理量读取最近 250 篇文章与 250 条未入库记录；任务运行时 3 秒仅轮询轻量 Job 快照，阶段变化或任务结束才刷新完整明细，空闲时 15 秒刷新，页面不可见时暂停周期请求。
- SQLite 在服务启动和部署脚本中统一初始化为 WAL、`synchronous=NORMAL`、`busy_timeout=5000`、外键开启，并执行 `PRAGMA optimize`；部署包同时排除 `-wal`/`-shm` 文件。
- 技术工作队列使用 5 秒进程内短缓存合并抓取记录和后台导航的重复扫描，Job 结束后主动失效；保持单实例语义，不引入额外状态表或外部缓存。
- 数据看板在页面不可见时暂停自动刷新，Job 历史读取设置最近 500 条硬上限，消除无限历史加载。
- Event 新增并持久化公开日期键与排序时间；公开列表先读取有限日期键，再查询这些日期的数据，每个日期最多 250 篇；列表查询不再读取正文，公开总数由数据库计数。
- 关键词候选由逐 phrase 的 `findUnique + upsert` 改为一次 `findMany` 预取和短事务写入；聚合浏览量统一在一个短事务中提交，减少 SQLite 往返和锁切换。
- Nginx 模板显式启用 JS/CSS/JSON/SVG/HTML gzip，并保留 `Vary`，减少公开端与后台接口传输体积。
- 采集阶段按单个数据源的本轮 URL 一次预取 Article 和 DiscardedItem，再进行内存分类；保留唯一约束兜底，消除每条文章两次重复只读查询。
- 公开详情导航只加载当前日期组，并在组边界额外查询相邻日期的一条 Event，不再为上一篇/下一篇读取全部公开 Event。
- Dashboard 统计使用 15 秒进程内 Promise 缓存合并重复请求，Job 完成后主动失效；在当前低频管理员场景下先避免重复聚合，不引入预计算表。
- Dashboard 触发的收件箱趋势快照按分钟限频并合并并发写入；人工归类操作仍即时记录，避免看板刷新制造无意义 SQLite 写锁。

仍未完成的结构性项目主要是：抓取记录 Source 明细按需加载，以及 Dashboard 的完整数据库聚合。Dashboard 已通过隐藏暂停、历史上限和 15 秒请求合并控制实际成本；采集 URL 和关键词候选的主要高频往返也已完成批量化。剩余两项属于未来数据量显著增长后的容量优化，不影响本轮文章打开与后台切换的主要交互收益。公开列表已经使用持久化排序字段和数据库日期条件，抓取记录高频轮询也不再重复传输完整明细。

本次为静态代码与配置审查，覆盖：

- 公开首页、详情页、公开 API、站点地图与访问计数；
- 管理后台情报收件箱、抓取记录、设置/数据看板；
- Prisma 查询、索引、SQLite 模式与批量写入；
- collect → process → cluster → ai → push 批处理链路；
- 缓存、轮询、Job 进度写入；
- Next.js 客户端边界、Nginx 静态资源与 PM2 单实例部署。

本次没有执行压力测试、生产构建体积分析或全量测试；文中的收益判断来自当前实际代码路径与复杂度分析。实施各阶段时应补齐基准数据。

## 3. 当前性能基线

### 3.1 已具备的有效优化

- `Article`、`Event`、日志与 Job 已有多组组合索引，关键批处理查询也有 `take: 500` 上限。
- collect/process/AI/push 均设置了并发上限、超时或退避，避免无限并发压垮源站、AI Provider 和 SQLite。

## 4. 主要发现与改进建议

### P0-1：公开读取必须从“全量加载后内存处理”改为数据库分页

**现状**

`src/lib/public-article-service.ts` 的 `loadPublishedEvents()` 会读取全部公开 Event 及代表文章：

- 首页：全量读取后在 Node.js 中搜索、排序、按日期分组、截取前 3/10 个日期；
- 加载更多：再次全量读取，再用 `before` 在内存中过滤；
- probe：再次全量读取，只为得到总数；
- 详情页：全量读取、排序后查找当前项、前后项和品牌相关项；
- `generateMetadata()` 与页面渲染分别调用详情服务，单次详情请求可能重复两轮全量扫描；
- sitemap 同样读取全部公开 Event，虽然 sitemap 本身允许全量，但不应与首页/详情共用同一重查询。

公开页面还被根布局和页面设置为 `force-dynamic`，因此每次页面请求都会进入服务端读取路径；60 秒进程内缓存只能部分掩盖问题，详情查询没有对应缓存。

**风险**

- 复杂度随公开 Event 数量近似 O(N log N)，且读取了 `cleanContent` 等较大字段；
- 多个搜索词、`before` 值会不断生成缓存 key，当前 Map 没有容量上限，过期条目也不会主动清理；
- 进程重启后缓存清空，无法稳定保证 TTFB；
- 搜索使用内存 `includes`，无法利用数据库索引。

**建议方案**

1. 建立明确的公开排序游标，建议在 `Event` 上持久化：
   - `publicSortAt`：代表文章 `publishedAt ?? firstSeenAt`；
   - 可选 `publicPinnedUntil`，或继续通过代表文章关联判断；
   - 公开状态变化、代表文章变化、发布时间变化时同步更新。
2. 首页改为数据库查询：
   - 先查询目标日期键或使用复合游标 `(publicSortAt, id)`；
   - 再按游标读取有限条 Event；
   - 列表查询不选择 `cleanContent`，新增/使用持久化 `publicExcerpt`，避免读正文再 `stripHtml`。
3. 搜索与普通列表分开：
   - 当前规模先使用 SQLite `contains` 查询 title/summary/brand，并设最多返回条数；
   - 当公开数据达到数万条且搜索成为热点，再评估 SQLite FTS5；当前不应提前引入外部搜索服务。
4. 详情按 `event.id` 单条查询；前后导航使用两条带范围条件的查询；相关内容用有限时间窗口/品牌候选查询，不再扫描全部公开 Event。
5. probe 改为轻量 revision：使用 `MAX(publicContentUpdatedAt)` + `COUNT(*)`，或维护一个公开 feed revision 设置值；不要为检查“是否有新文章”读取所有行。
6. 将公开缓存改为容量受控缓存：TTL + 最大 key 数 + 访问时清理过期项。若完成数据库游标分页，缓存只作为锦上添花，不作为性能正确性的前提。
7. 评估移除公开路由的 `force-dynamic`：
   - 可使用短时 `revalidate` 或标签失效；
   - 后台发布/撤回时触发失效；
   - 管理后台继续保持动态。

**验收指标**

- 首页/加载更多查询返回行数与本页展示量同阶，不随全库 Event 数增长；
- 详情页不调用 `loadPublishedEvents()`；
- 10,000 个公开 Event 下，首页 API P95 < 200 ms（同机 SQLite、无网络）；
- 列表接口单次响应体建议 < 200 KB；
- 缓存 key 数有硬上限。

### P0-2：抓取记录从“3 秒全量快照”改为分层快照与按需加载

**现状**

`CrawlLogTab` 以 `limit: 500` 调用 `useCrawlLogSnapshot()`，可见页面每 3 秒请求 `/api/crawl-log/status`。服务端每轮会：

- 查询 Job、最多 500 篇 Article、最多 500 条 DiscardedItem、全部 Source；
- 额外调用技术工作队列，扫描失败 Article、全部活动 Event，并读取相关 PushLog；
- 拼装每个 Source 下的文章和丢弃记录；
- 返回完整 JSON，前端替换整份 snapshot，再重新筛选、计数和渲染。

这条链路在 Job 运行与空闲时使用同样频率、同样数据量。

**风险**

- 500 + 500 行每 3 秒持续读取与传输，单个后台页面每天可产生 28,800 次请求；
- 多个管理员标签页会倍增负载；
- SQLite 与后台 Job 共用单文件，轮询会增加读写竞争；
- 前端隐藏/折叠的 Source 仍接收完整明细。

**建议方案**

将接口拆为三个层级：

1. **Job 轻快照**：Job 状态、阶段、进度、心跳、技术问题总数、Source 摘要；运行时 2~3 秒轮询，空闲时 15~30 秒轮询。
2. **Source 明细**：按 `sourceId + cursor + filters` 请求 Article/DiscardedItem，只有展开 Source 或筛选命中时加载。
3. **详情接口**：继续按需读取单篇流程详情，不进入轮询响应。

补充规则：

- 响应包含 `revision`（可用 Job.updatedAt、最新 Article/DiscardedItem.createdAt、技术队列更新时间组合）；revision 未变化返回 `304` 或极小响应；
- 页面不可见时暂停 interval，而不是仅在恢复可见时补拉；
- Job 结束后立即刷新一次，随后降频；
- Source 列表使用服务端分页/游标，不把 500 条作为长期上限模型；
- 如果未来确需实时推送，优先使用单实例内的 SSE 只传“revision changed”，客户端再拉轻快照，不通过 SSE 传完整列表。当前先做自适应轮询即可。

**验收指标**

- 空闲状态每分钟最多 2~4 个轻请求；
- Job 运行时轮询响应体 < 20 KB；
- 文章/丢弃明细不随 Job 轮询重复传输；
- 页面 hidden 后不产生周期请求；
- 10,000 Article 下抓取记录首屏查询 P95 < 300 ms。

### P0-3：统一初始化 SQLite WAL 与连接参数

**现状**

当前 `src/lib/db.ts` 只创建 PrismaClient，没有执行 SQLite 连接初始化。本地实际查询显示：

- `journal_mode=delete`；
- `busy_timeout=5000`；
- `synchronous=FULL`（Pragma 值 2）。

而 `collect.ts` 注释明确按“SQLite WAL 模式”分析写入竞态，代码假设与运行事实不一致。

**风险**

- delete journal 下写事务更容易阻塞公开读取和后台 API；
- 公开详情浏览计数、Job 心跳、批处理写入、管理员操作可能在同一时段竞争；
- 仅在某次手工连接设置 PRAGMA 不够，必须确保生产数据库与新连接都生效。

**建议方案**

1. 新增显式数据库初始化脚本，在部署/首次建库后执行并验证：
   - `PRAGMA journal_mode=WAL;`
   - `PRAGMA synchronous=NORMAL;`
   - `PRAGMA busy_timeout=5000;`（可保留并根据实测调整）；
   - `PRAGMA foreign_keys=ON;`
2. 服务启动时做只读/幂等校验，发现 journal mode 不是 WAL 时输出明确告警；不要在每个请求中执行 PRAGMA。
3. 部署文档增加 `db:optimize` 或初始化命令，并在迁移后运行 `PRAGMA optimize`。
4. 保持 PM2 单实例。WAL 改善读写并发，但不会让 SQLite 适合多进程高写入。
5. VACUUM 继续只允许人工执行，并在无 Job 运行时进行。

**验收指标**

- 开发与生产启动日志确认 `journal_mode=wal`；
- Job 运行期间公开列表并发读取不出现 `database is locked`；
- WAL/SHM 文件的备份与运维说明同步更新。

### P1-1：拆分管理后台路由和客户端 bundle

**现状**

`src/components/admin-shell.tsx` 静态导入 `IntelligenceInbox`、`CrawlLogTab`、`SettingsTab`。`SettingsTab` 又静态导入数据看板、源管理、关键词、AI 模型、提示词、推送、数据清理等全部子模块。

虽然 React 只渲染当前 tab，但这些模块会进入同一管理后台客户端依赖图。当前较大的客户端源码包括：

- `intelligence-inbox.tsx`：约 81 KB；
- `dashboard-tab.tsx`：约 49 KB；
- `crawl-log-tab.tsx`：约 30 KB；
- `source-management.tsx`：约 22 KB；
- 多个设置子模块约 10~17 KB。

**建议方案**

优先采用真实路由，而不是继续扩大 `?tab=` 单页面：

- `/admin/inbox`
- `/admin/crawl-log`
- `/admin/settings/[section]`

每个路由有独立 Server Page 和客户端岛，天然代码分割，并保留现有深链参数。根 `/admin` 重定向到收件箱。

如果短期不改路由，至少对三个顶层 tab 以及设置的重型 section 使用 `next/dynamic` 懒加载，并为每个模块提供 skeleton。

同时：

- 将 `PublicSharePoster` 动态导入；只有用户点“分享”时才需要 `qrcode` 和 Dialog 逻辑；
- 检查 `SettingsTab` 中 Radix Tabs 是否会同时挂载所有内容，确保未激活 section 不发请求；
- 把 80 KB 的收件箱按“列表、详情、聚类校准、批量操作”拆成稳定子组件和 hooks，配合 memo，但拆文件本身不是性能目标，减少无关重渲染才是目标。

**验收指标**

- 生产构建中 `/admin/inbox` 不包含 dashboard、source management、prompts 等非当前页面 chunk；
- 后台首屏 First Load JS 较改造前下降至少 30%；
- 切换页面时按需加载且保留可感知的 loading 状态。

### P1-2：文章列表 Facet 不应在每次翻页/搜索时全表重算

**现状**

`listArticles()` 每次列表请求并行执行：

- 当前筛选下的分页查询；
- 当前筛选总数；
- 全库 category groupBy；
- 全库所有非空 brand 行读取，然后在 Node.js 中拆分品牌并计数。

因此用户每次翻页、改变排序或输入搜索词，都会重新扫描品牌列；品牌数量与 Article 数增长后，这会成为收件箱的固定额外成本。

**建议方案**

1. 把 facets 从列表响应拆到 `/api/articles/facets`；后台首次进入或数据 revision 变化时加载。
2. 对 facets 做 1~5 分钟 TTL 缓存，并在 Article AI/人工品牌修改、删除、重建后失效。
3. 中期将品牌规范化为关联表或维护 `ArticleBrand`，使筛选和计数可走索引；当前品牌用字符串存多个值，数据库无法高效统计。
4. 删除未被调用的 `all=true` 无上限列表能力，避免后续误用导致一次返回全表。
5. 对 `skip/offset` 深分页设上限；数据上万后改 `(createdAt, id)` 游标分页。

**验收指标**

- 翻页只执行 items + count，不再读取全库 brand；
- facets 响应有缓存命中指标；
- 收件箱常规列表 P95 < 200 ms。

### P1-3：技术工作队列与推送状态查询应避免反复扫描历史 PushLog

**现状**

`getTechnicalWorkQueue()` 查询全部失败 Article 和全部活动 Event，然后 `getPushTargetStatesForEvents()` 读取这些 Event 对应 Webhook 的全部 PushLog，再在内存中挑最新一条。

该服务既用于抓取记录快照，也用于后台导航的工作队列计数。随着 Event 和 PushLog 增长，成本会持续上升。

**建议方案**

1. 最优方案：新增 Event/Webhook 当前投递状态表，例如 `EventPushTargetState`，主键 `(eventId, webhookUrlHash)`，每次投递 upsert 当前状态；`PushLog` 继续作为审计历史。
2. 轻量过渡方案：先只查询“可能失败”的 Event，使用 SQL 子查询/window function 取每个 `(eventId, webhookUrl)` 最新记录，不把全部历史日志拉回 Node.js。
3. 导航计数接口使用短 TTL（5~15 秒）并由相关写操作失效；切 tab 不必每次执行完整队列扫描。
4. 抓取记录轻快照直接返回服务端聚合计数，不需要返回全部技术项后再计算总数。

**验收指标**

- 工作队列计数查询成本与当前失败目标数相关，而不是与全部历史 PushLog 相关；
- 100,000 PushLog 下工作队列摘要 P95 < 200 ms。

### P1-4：批处理中的高频小查询和逐行写入需要合并

#### 4.1 collect 单条多次查询

每个列表 item 依次执行 Article URL 查询、DiscardedItem URL 查询、Article create/update，属于典型 N×2~3 数据库往返。Source 之间并发 4，但同 Source item 串行。

建议按 Source 批次预取：

- 一次查询本批全部 normalized URL 对应的 Article；
- 一次查询本批全部 discarded URL；
- 内存分类后用事务分批创建/更新；
- 保留唯一约束处理竞态。

#### 4.2 关键词候选每篇最多产生大量小写入

`recordKeywordCandidates()` 每个 phrase 先 `findUnique` 再 `upsert`，每篇标题最多 24 个 phrase，即最多 48 次数据库操作。关键词未命中多时，会放大 SQLite 写锁占用。

建议：

- 一次 `findMany({ phrase: { in: phrases } })`；
- 在一个短事务内完成 upsert；
- 对单次 process batch 汇总相同 phrase，再统一落库；
- 候选只需近似统计时，可减少 2~6 字全组合数量，优先品牌词/专名抽取。

#### 4.3 发布快照逐行同步 Event

`refreshPublicPublications()`、`rebuildPublicPublicationSnapshot()` 对每个 Article 逐行 update，并为每篇再查 Event、更新 Event。设置权重变化时还会先逐篇更新分数，再逐篇重建发布状态，长事务最长允许 120 秒。

建议：

- 根据新规则把 Article 分成 published/revoked/unpublished 三组，使用 `updateMany` 写公共字段；
- 仅对代表文章关联的 Event 做批量查询与批量更新；
- 评分重算与发布重建分批（如 200 条/批），或使用单条 SQL CASE 计算；
- 不要在一个超长 SQLite 写事务里逐篇更新全库。设置保存可先写入一个 rebuild Job，再由单实例 worker 执行，后台显示进度。

#### 4.4 Event 聚类候选加载过宽

每篇待聚类文章会读取时间窗口内全部活动 Event，并为每个 Event 加载最多 8 篇文章，再在内存排序后取少数候选。文章量和窗口内 Event 数增加后，接近 O(待聚类数 × 活动 Event 数)。

建议：

- 为 Event 持久化代表 `eventKey`/规范化主题键和必要的轻量指纹；
- 候选查询先按 `eventKey` 精确/前缀、时间窗和代表标题过滤，并设置数据库 `take`；
- 只有规则候选不足时再扩大到近 N 个 Event；
- 将 AI event key 生成与 AI 内容分析的模型调用关系纳入成本监控，避免聚类阶段额外请求成为吞吐瓶颈。

**验收指标**

- collect 每 100 条 item 的 SQL 次数显著下降（目标 < 250 次，后续继续优化）；
- 单篇关键词未命中不再产生 40+ 次数据库往返；
- 10,000 Article 全量发布重建不持有分钟级写事务；
- 单篇聚类候选加载行数有硬上限。

### P1-5：数据看板改为数据库聚合，Job 历史必须服务端分页

**现状**

`getDashboardAnalytics()` 会读取时间窗口内的全部 Article、DiscardedItem、FetchLog 到 Node.js 聚合；另一个 `recentJobs` 查询没有 `take` 和时间范围，然后在内存中过滤与分页。数据看板开启后每 30 秒自动刷新。

**风险**

- 30 天窗口数据越多，响应时间、内存和 JSON 构造成本越高；
- Job 即使只显示一页，也会读取全部历史；
- 当打开 Source 详情时还会并行执行第二套相似统计。

**建议方案**

- 使用 SQL `GROUP BY sourceId, date(createdAt)` 聚合 Article/Discarded/FetchLog；
- Job 筛选条件下推数据库，并执行 `count + skip/take`；
- 将总览、Source 详情和 Job 列表拆成独立接口，避免查看一部分却重复获取全部数据；
- 自动刷新只刷新今日轻量指标和运行中 Job，历史趋势可缓存 1~5 分钟；
- 页面 hidden 时暂停 30 秒刷新。

**验收指标**

- Dashboard 单次查询返回聚合行，而非全部明细行；
- Job 查询始终有 `take`；
- 30 天、100,000 Article 下概览 P95 < 500 ms。

### P2-1：按查询模式补索引，但必须以 EXPLAIN 结果为准

当前索引基础不错，但以下查询模式可能缺少匹配索引：

- 公开 Event：`(status, publicStatus, publicSortAt DESC, id)`；
- Event 聚类：`(status, eventKey, lastSeenAt DESC)`（需先新增 Event.eventKey）；
- Article Source 时间线：`(sourceId, publishedAt DESC, createdAt DESC)`；
- Dashboard：`(sourceId, createdAt)` 或按真实筛选组合调整；
- Job 历史：`(type, createdAt DESC)`，若常按完成状态查询可考虑 `(type, status, createdAt DESC)`；
- 关键词候选：现有 `(status, occurrences)` 与 `updatedAt` 排序不完全一致，可根据真实查询决定是否扩展。

不要机械地增加所有组合索引。SQLite 每个索引都会增加写放大和文件体积；实施前后应使用 `EXPLAIN QUERY PLAN` 验证是否命中。

### P2-2：浏览量与原文点击计数进一步批量化

浏览量已经按 1 秒合并，但 flush 中仍按 Article 逐条 `UPDATE`；原文点击则每次请求直接写库。

建议：

- flush 使用一个短事务，或按批次构造 SQL；
- 原文点击也进入同一类内存聚合队列；
- 进程退出丢失少量计数是可接受的，需在文档明确“近似统计”；若要求严格计数，才考虑持久队列。

### P2-3：缓存治理与一致性

当前缓存均为单进程内存缓存，适合 PM2 单实例，但需要统一约束：

- `publicArticleListCache` 增加容量上限和过期清理；
- facets、工作队列摘要、Dashboard 历史趋势可复用通用 TTL/LRU；
- 记录 hit/miss/entry count，避免缓存无效却无人发现；
- 继续保持单实例，不能把内存缓存当成多实例一致性方案；
- 若未来改多实例，应先外置 scheduler/job ownership，再讨论共享缓存。

### P2-4：响应压缩与静态资源策略

Nginx 已正确直接提供 hashed Next 静态资源，但模板未显式体现 gzip/Brotli。应确认服务器全局是否已开启；若没有：

- 为 JS/CSS/JSON/SVG 开启 gzip；可用时启用 Brotli；
- 动态 API 不要设置长期缓存；
- 公开 GET API 可在完成 revision/失效模型后使用短 `s-maxage` + `stale-while-revalidate`；
- 保留 `/_next/static/` 的 `immutable`。

## 5. 不建议当前实施的方案

以下方案在当前规模和架构下投入大于收益：

- **Redis 缓存**：当前单实例，先修复数据库全量查询和无界缓存即可。
- **RabbitMQ/Kafka/独立 Worker**：现有 Job 单实例模型已能满足低吞吐任务；先减少逐行 SQL 和长事务。
- **立即迁移 PostgreSQL**：SQLite 的主要问题尚未被 WAL、查询下推和批量写优化充分处理。达到持续高并发写、多实例或数据库明显超出单机能力时再迁移。
- **React 全面 memo 化**：后台 bundle 和接口全量数据问题优先级更高。没有 profiler 证据时，大范围 `memo/useMemo` 只会增加复杂度。
- **重新引入已删除的 SSE 进度接口**：当前应先拆轻快照与自适应轮询；SSE 只在轮询成本仍不可接受时用作变更通知。
- **为了性能保留历史数据兼容分支**：项目约束为重新获取新数据，Schema 改造可直接以新数据模型为准，避免双写与兼容查询拖慢主路径。

## 6. 推荐实施顺序

### 阶段 A：建立可测基线（P0，0.5~1 天）

- 增加生产构建 bundle 报告；记录公开首页、详情、抓取记录、收件箱、Dashboard 的响应时间与响应体大小。
- 准备 1k / 10k / 100k Article+Event 的可重复 seed 脚本，只用于性能环境。
- 为关键服务增加轻量耗时日志：query/build/serialize 分段，慢于阈值才输出。
- 使用 `EXPLAIN QUERY PLAN` 保存关键查询基线。

### 阶段 B：公开读取与 SQLite（P0，2~4 天）

- 引入公开排序字段/摘要字段及索引；不保留旧数据兼容，重新采集/重建。
- 重写公开列表、详情、probe 为有限数据库查询。
- 配置并验证 WAL、synchronous、busy timeout、optimize。
- 为公开缓存加容量治理，评估短时 revalidate。

### 阶段 C：抓取记录轻量化（P0/P1，2~4 天）

- 拆 Job 轻快照、Source 明细、详情接口。
- 运行中高频、空闲低频、hidden 暂停。
- 服务端分页和 revision/304。
- 技术工作队列先做轻量聚合；随后视 PushLog 规模决定是否新增当前状态表。

### 阶段 D：后台拆包与列表 Facet（P1，2~3 天）

- 后台改真实子路由或顶层动态导入。
- Settings section 懒加载。
- Facet 独立接口 + TTL/失效。
- 动态加载分享海报模块。

### 阶段 E：批处理与 Dashboard（P1，3~6 天）

- collect URL 批量预取；关键词候选批量写。
- 发布重建与评分重算批处理/Job 化。
- Event 候选查询收窄并设置硬上限。
- Dashboard 改 SQL 聚合，Job 历史数据库分页。

### 阶段 F：容量复验（1~2 天）

- 以 10k 和 100k 数据重新测量；
- 检查 SQLite 锁等待、WAL 文件增长、内存峰值、Node event loop delay；
- 只有仍存在明确瓶颈时，才决定 FTS5、共享缓存或数据库迁移。

## 7. 建议的性能门禁

建议在后续改造文档和 PR 中固定以下门禁：

| 场景 | 建议目标 |
| --- | --- |
| 公开首页 SSR/API（10k Event） | P95 < 200 ms |
| 公开详情（10k Event） | P95 < 150 ms，不扫描全表 |
| 收件箱普通分页（100k Article） | P95 < 200 ms |
| 抓取记录 Job 轻快照 | P95 < 100 ms，响应 < 20 KB |
| 抓取记录 Source 明细 | P95 < 300 ms，服务端分页 |
| Dashboard 30 天（100k Article） | P95 < 500 ms |
| 后台首屏 First Load JS | 较当前生产基线下降 ≥ 30% |
| SQLite | WAL；常规运行无 `database is locked` |
| 长事务 | 常规管理操作 < 2 秒；全量重建必须 Job 化并显示进度 |

## 8. 实施时的代码边界

- 公开可见性仍由 `src/lib/public-article-service.ts` 和发布快照服务负责；不要把规则搬回组件或 Route Handler。
- Event 仍是公开和推送唯一去重门；性能改造不能恢复 Article 级推送/公开双事实源。
- 抓取记录只负责 Job/技术恢复；情报收件箱只负责内容和 Event 人工校准。接口拆分不能重新制造职责重复。
- `runJob()` 仍是批任务入口；不要为了吞吐绕开 Job/Mutation Guard 私自启动并行 worker。
- 结构性改造同步更新 `README.md`、`AGENTS.md`、Prisma migration 与 `bat/` 运维说明。
- 不兼容历史数据；新增公开排序字段、Event 候选键或推送当前状态表时，可重新下载/采集并直接使用新结构。
