# Hot2-S 项目边界收敛与性能优化实施计划

> 文档定位：基于 2026-07-19 项目代码审查形成的实施交接文档。  
> 本文只定义目标、边界、顺序和验收标准，不代表任何改造已经完成。后续实施者必须结合实施时的最新代码重新核对，不得机械照搬本文中的文件位置或现状描述。

## 1. 背景与结论

项目当前技术路线基本合理：Next.js、Prisma、SQLite 和单进程 Job 模型与现阶段规模匹配；Event 已基本成为公开展示和推送的业务单位；Service、DTO、短缓存、适应性轮询及 SQLite WAL 等轻量措施也符合项目规模。

本轮不进行推倒重来，核心目标是继续收敛以下四类边界：

1. 同一业务状态只能有一个权威事实源。
2. Route Handler 只负责协议适配，业务写入和事务必须进入 Service。
3. 批处理必须限制每轮数据库读取和写入规模，而不只是把全量数组放到内存中切片。
4. 昂贵计算必须放在廉价、确定性筛选之后，只用于解决真正的歧义。

当前最高风险是：**Event 的 `pushedAt` 与当前启用目标的最新 PushLog 同时参与投递判断，存在状态分叉并导致静默漏推的可能。**

## 2. 改造原则

### 2.1 必须遵守

- 目标只取“能跑起来、能维护、性能好”的平衡点。
- Event 是公开和推送的唯一业务门禁；`Event.publicStatus` 是公开事实。
- PushLog 中“当前启用目标的最新投递结果”是目标级投递事实。
- Article 继续承担原始内容、AI 结果和人工校准记录，不重新扩张为公开或推送事实源。
- `needs_review` 可以继续 AI 分析，但不能成为代表文章、公开或推送。
- 新改造按重新获取新数据处理，不为旧数据增加兼容层；如需调整 Schema，创建正常迁移，但不编写历史数据兼容逻辑。
- 复用现有 Service、Contract、设置目录和轻量缓存，不重复建立平行模块。
- 功能或结构性变更必须同步更新 `README.md`；`AGENTS.md` 仅在长期架构约束变化时更新。
- 每个工作流必须独立完成，计划文档本身不能作为实施完成证明。

### 2.2 明确不做

- 不引入 Redis、消息队列、微服务或独立 Worker 集群。
- 不引入向量数据库或 Embedding 召回。
- 不引入通用 Repository、复杂领域框架或前端全局状态框架。
- 不进行全项目重写。
- 不为旧 `JobType`、旧 Token 流程、旧函数签名或旧状态格式继续增加兼容分支。
- 不以“预防未来超大规模”为理由过度设计。

## 4. 目标架构与事实归属

| 业务事实 | 唯一权威来源 | 可存在的派生投影 | 禁止做法 |
| --- | --- | --- | --- |
| Event 是否公开 | `Event.publicStatus` | 代表 Article 的公开结果投影 | 由 Article 单独决定 Event 是否公开 |
| Event 代表内容 | `Event.representativeArticleId` | Article 列表中的代表标记 | UI 或 Route 内自行推断代表文章 |
| 目标级推送结果 | 当前启用 Webhook 对应的最新 PushLog | `Event.pushedAt`、重试时间、重试次数 | 仅凭 `Event.pushedAt` 判断是否还需投递 |
| Article 技术故障 | 各流水线阶段状态、重试次数和下一重试时间 | 技术工作队列 DTO | 用一个模糊字段同时表达隐藏、放弃和流水线禁用 |
| 公开快照刷新 | `public-publication-service` | Article 代表投影、公开缓存 | 设置、Route 或组件各自重算公开规则 |
| 调优建议与统计快照 | 对应写入 Service 或 Job/审核业务事件 | GET DTO 与短缓存 | 轮询 GET 隐式写数据库 |

## 6. 工作流一：公开分享海报接口收口（P0）

### 6.1 当前问题

- `src/components/public-share-poster.tsx` 匿名调用 `POST /api/public/share-poster`。
- `src/proxy.ts` 当前只明确放行公开文章读取及 view/click POST，海报 POST 可能被生产鉴权拦截。
- 客户端提交标题、摘要、发布时间和任意分享 URL，服务端会直接生成图片，边界过宽。
- 海报生成包含二维码和 Sharp 图片处理，属于相对昂贵的匿名计算接口。

### 6.2 目标状态

- 匿名用户可以为真实、当前可公开的 Event 生成分享海报。
- 客户端只提交 Event ID；标题、摘要、发布时间和正式分享 URL 由服务端读取或构造。
- 请求必须通过现有公开可见性 Service 校验，不能为未公开 Event 生成海报。
- 只精确放行该 POST，不扩大其他 `/api/public/*` 权限。
- 使用符合当前单实例规模的简单限流和输入上限。

### 6.3 实施任务

- [ ] 修改海报请求契约，仅接受 `eventId`。
- [ ] 在服务端复用公开详情/公开可见性读取能力，取得代表 Article 的标题、摘要和发布时间。
- [ ] 使用 `NEXT_PUBLIC_SITE_URL` 及 Event ID 在服务端构造站内分享 URL，不信任客户端任意 URL。
- [ ] 在 `src/proxy.ts` 精确允许 `POST /api/public/share-poster`，其他方法返回 405。
- [ ] 增加单实例内存级、固定窗口的简单限流；限制 key 数量并定期淘汰，避免无界 Map。
- [ ] 对不存在、已撤回或不可公开 Event 返回稳定的 404/403 类响应。
- [ ] 同步更新 README 中公开接口和分享海报说明。

### 6.4 验收标准

- [ ] 未登录用户可为公开 Event 生成海报。
- [ ] 未公开、已撤回、无代表文章的 Event 不能生成海报。
- [ ] 客户端不能借接口生成任意标题、摘要或外部 URL 的海报。
- [ ] 只新增一个精确公开 POST 例外，没有放宽后台 API。

## 7. 工作流二：统一 Event 投递事实（P0）

### 7.1 当前问题

- `src/lib/push/delivery.ts` 已能读取当前启用 Webhook 的最新 PushLog，并据此选择未成功目标。
- `src/lib/push/policy.ts` 的自动候选查询仍以 `Event.pushedAt: null` 为前置条件。
- `src/lib/event-service.ts` 合并 Event 时会把源 Event 的 `pushedAt` 直接复制给目标 Event。
- 新增 Webhook、启停目标、Event 合并、代表文章变化或部分投递成功后，`pushedAt` 可能与目标级真实收据不一致。

### 7.2 目标状态

- 当前启用 Webhook 的最新 PushLog 是是否还需投递的唯一目标级事实。
- `Event.pushedAt` 只表示“按当前启用目标完成全部投递”的派生完成时间。
- 任一启用目标为 `never_attempted` 或最新为 `failure` 时，普通自动推送仍可选中该 Event。
- Event 合并、拆分、代表文章变化、Webhook 配置变化和每次投递后，都能按同一规则校准派生状态。
- `normal`、`retry_failed`、`repush_all` 三种模式语义保持不变。

### 7.3 推荐边界

在推送领域 Service 中集中提供类似能力，名称可根据现有代码调整：

```ts
getEventDeliveryState(eventId)
reconcileEventDeliveryState(eventId)
reconcileEventDeliveryStates(eventIds)
```

建议统一派生：

- `completed`：至少有一个启用目标，且所有目标最新状态均为 success。
- `pending`：存在 `never_attempted`。
- `failed`：存在 failure，且已达到人工处理条件或没有未来重试时间。
- `retry_waiting`：存在 failure，且仍有未来重试时间。
- `disabled`：没有启用目标；不伪造完成或失败收据。

### 7.4 实施任务

- [ ] 搜索所有读取或写入 `Event.pushedAt`、`nextPushRetryAt`、`pushRetryCount` 的位置，列出调用矩阵。
- [ ] 将“当前启用目标的最新 PushLog”收敛为单一查询/派生函数。
- [ ] 修改自动推送候选选择：不能仅以 `pushedAt: null` 筛选；数据库先筛业务资格，再批量计算目标缺口。
- [ ] 推送结束后统一调用 reconcile，禁止各分支自行拼装派生状态。
- [ ] Event 合并时不复制源 `pushedAt`；明确处理 PushLog 归属：保留历史审计，同时按目标 Event 当前收据重新派生状态。
- [ ] Event 拆分和代表文章切换后重新校准目标 Event；历史 PushLog 的代表文章快照不得被改写。
- [ ] Webhook 设置改变后，使受影响 Event 能重新进入缺失目标投递判断；不要逐 Event 长事务同步发送。
- [ ] 技术工作队列和后台显示复用同一投递状态，不再各自解析失败。
- [ ] 同步 README 中 `PushLog`、`pushedAt`、Webhook 变更和 Event 合并后的语义。

### 7.5 必要回归场景

- [ ] 从未投递的 Event：所有启用目标均需发送。
- [ ] 部分成功：普通推送只补失败或未尝试目标。
- [ ] 全部成功：普通推送不重复发送。
- [ ] 新增一个启用 Webhook：既有 Event 能识别新目标缺少收据。
- [ ] 禁用失败 Webhook：按剩余启用目标重新派生完成状态。
- [ ] 已推送 Event 合并到未推送 Event：不能因复制时间而漏推。
- [ ] 已推送 Event 更换代表文章：历史日志保持快照，新的业务语义按产品规则明确处理。
- [ ] `retry_failed` 只恢复最新失败目标；`repush_all` 明确重推全部启用目标。

## 8. 工作流三：技术异常终态收口（P0）

### 8.1 当前问题

- 当前工作树正在引入 `fetchRetryCount`、`nextFetchRetryAt`、`technicalIgnoredAt`、`pushRetryCount` 等字段。
- `src/app/api/articles/[id]/technical-status/route.ts` 直接写 Prisma，绕过 Service 和 `runExclusiveMutation`。
- README 将“忽略”描述为不再计入任务中心待办，但流水线查询也使用 `technicalIgnoredAt: null`，实际含义可能变成停止后续处理、聚类、AI 和推送。
- 一个 Article 可能同时存在多个阶段故障；单个合并后的 `retryAvailableAt` 和 `state` 容易丢失阶段差异。
- Schema、迁移和 Prisma Client 在实施时必须保持一致。

### 8.2 先做语义决策

默认采用最小、清晰的语义：

- **忽略技术待办**：只影响后台技术队列展示，不改变流水线资格。
- **永久放弃处理**：如果确有产品需求，使用语义明确的独立终态，例如 `abandonedAt/abandonedReason`，不能复用 `technicalIgnoredAt`。

如果项目不需要“永久放弃”，则不要新增该能力。

### 8.3 目标状态

- 技术状态变更全部由 Service 负责，并通过写入门禁执行。
- 各阶段分别表达：失败状态、重试次数、下一重试时间和是否耗尽。
- 队列 DTO 可以聚合展示，但不能覆盖或模糊底层阶段事实。
- `manual` 与 `auto_retry` 判断必须按每个问题计算；一个阶段等待重试不能掩盖另一个阶段已经耗尽。
- Job 完成、人工恢复或状态修改后统一失效技术队列缓存。

### 8.4 实施任务

- [ ] 新建或复用技术恢复 Service，Route 只解析 `ignore/restore` 并调用 Service。
- [ ] Service 使用 `runExclusiveMutation` 或由调用方统一进入写入门禁；避免 Route 直接 Prisma 写入。
- [ ] 从 process、cluster、AI、push 资格查询中移除“仅用于隐藏待办”的 `technicalIgnoredAt` 门禁。
- [ ] 将技术队列 DTO 改为阶段级明细，例如每个 issue 自带 `state` 和 `retryAvailableAt`。
- [ ] 聚合 Article 显示状态时，只要任一阶段为 manual，整体就不能显示成纯 `auto_retry`。
- [ ] 明确抓取/正文处理命名，避免 `fetchStatus` 却展示为含义不一致的 `process_failed`。
- [ ] Prisma Schema 与迁移按最终语义整理；不保留试验阶段的冗余字段。
- [ ] 重新生成 Prisma Client。
- [ ] 更新 README 中“忽略”“恢复”“自动重试”“人工处理”的准确含义。

### 8.5 必要回归场景

- [ ] 忽略一个技术待办后，任务中心不再显示，但下一次符合条件的流水线仍可处理。
- [ ] 恢复后技术待办重新出现，且缓存立即失效。
- [ ] 同时存在 AI 自动重试和聚类人工故障时，UI 能分别显示，不被合并成单一自动恢复状态。
- [ ] 达到重试上限后不会无限自动重试。
- [ ] Route Handler 不再直接更新 Article 技术状态。

## 9. 工作流四：让 GET 保持纯读取（P1）

### 9.1 当前问题

- `GET /api/feedback` 会尝试执行 `generateTuningSuggestions()` 并写数据库。
- Dashboard/analytics 读取链路需要继续核对 `InboxSnapshot` 的写入位置；若读请求触发快照写入，应一并移出。
- 周期轮询 GET 隐式写入会与 Job 和人工操作争抢 SQLite 单写者资源，也让缓存和错误语义变复杂。

### 9.2 目标状态

- GET API 只读取，不创建、更新或删除业务数据。
- 调优建议在明确业务事件后生成，例如审核完成、Job 完成或显式 POST。
- InboxSnapshot 在 Job 完成或待办事实发生关键变化时写入，Dashboard GET 只消费结果。
- 写入失败不能阻断只读页面返回已有数据。

### 9.3 实施任务

- [ ] 将 `generateTuningSuggestions()` 从 feedback GET 移到明确触发点或独立 POST。
- [ ] 搜索所有 GET Route 的直接和间接写操作，包括 `create/update/upsert/delete`、审计写入和快照写入。
- [ ] 将 InboxSnapshot 记录集中到一个幂等 Service，并在 Job 完成等低频事件调用。
- [ ] 保留必要的短缓存，但缓存刷新不能通过业务写入完成。
- [ ] 更新 README 中调优建议和快照的生成时机。

### 9.4 验收标准

- [ ] 重复请求 feedback、dashboard 和 analytics GET 不改变数据库业务行。
- [ ] 即使写入门禁被 Job 占用，读接口仍能稳定返回已有数据。
- [ ] 建议和快照仍能在明确触发点按预期更新。

## 10. 工作流五：公开快照批处理与短事务（P1）

### 10.1 当前问题

- `src/lib/public-publication-service.ts` 的单条同步会额外查询 Event，再分别更新 Article 和 Event。
- 批量刷新、来源开关刷新和全量重建仍逐 Article 调用该流程。
- 设置权重变化会逐篇重新计算评分，再逐篇重建公开快照，并放入最长 120 秒事务。
- SQLite 长写事务会阻塞其他写操作；全量读取和逐行 update 也会随数据线性增长。

### 10.2 目标状态

- 公开状态以 Event 和当前代表 Article 为批处理单位。
- 一批查询取得 Event、代表 Article、来源及公开判断所需字段。
- 非代表 Article 的公开投影使用批量 `updateMany` 清理。
- 只更新状态真正变化的 Event/Article，避免每次刷新都写 `evaluatedAt` 导致无意义更新。
- 大规模重算按稳定游标分块，每批使用短事务。
- 设置保存与全量重建不再依赖一个 120 秒事务维持“全有或全无”；允许显式记录重建中状态或同步完成小规模设置写入后分批重建。

### 10.3 推荐实现方式

- 使用 `(createdAt, id)` 或唯一 `id` 作为稳定游标，禁止 offset 深分页。
- 每批只读取有限 Event，包含代表 Article 所需字段。
- 先在内存中计算 next state，再按状态分组执行有限数量的 `updateMany`；确需不同时间戳的行才逐行更新。
- 每批单独短事务，批次完成后再失效一次公开缓存。
- 来源公开开关只定位受影响 Event，不扫描来源下所有历史非代表 Article。

### 10.4 实施任务

- [ ] 绘制公开刷新调用矩阵：AI 完成、人工编辑、代表切换、Event 合并拆分、来源开关、评分权重和公开设置变化。
- [ ] 新增 Event 维度批量刷新入口，逐步替换逐 Article 全量循环。
- [ ] 批量预取 Event 与代表 Article，消除 `syncCandidate()` 中的 N+1 Event 查询。
- [ ] 非代表投影按 Event 批量清理。
- [ ] 设置权重重算按数据库真分块执行，并复用同一公开批量刷新入口。
- [ ] 避免把设置保存和全库重算锁在一个 120 秒事务中。
- [ ] 来源公开开关只刷新受影响 Event。
- [ ] 保持 `Event.publicStatus` 唯一公开事实不变。
- [ ] 更新 README 的快照重建和运维说明；如新增运维脚本，同步 `package.json` 和部署说明。

### 10.5 验收标准

- [ ] 全量重建时单批查询和单事务行数存在明确上限。
- [ ] Event、代表 Article、非代表 Article 的公开投影满足现有不变量。
- [ ] 规则未变化的行不会被无意义重复写入。
- [ ] 来源开关、评分权重、人工公开策略和代表切换仍能正确刷新公开结果。
- [ ] 公开缓存只在实际变更后按批次或任务结束合理失效。

## 11. 工作流六：流水线数据库真分块（P1）

### 11.1 当前问题

- process、AI 等阶段会先读取全部待处理 ID，再在内存中按 `MAX_BATCH_SIZE` 切片。
- push 候选可能一次读取较多 Event 后在 JavaScript 中排序或补充状态。
- 这能限制单批业务执行，却不能限制最初数据库读取、内存占用和首响应时间。
- 批次大小应是数据库读取上限，而不是对全量结果的二次切片。

### 11.2 目标状态

- 每次数据库查询使用稳定排序、`take` 和游标，只读取一个批次。
- 每个阶段持续查询，直到当前 eligible backlog 被排空，Job 才能完成。
- 同一失败项必须通过状态/重试时间退出当前 eligible 集合，避免无限循环。
- Job 的 total 可以是运行开始时快照，processed/errors 则反映真实处理；文档和 UI 不假装 total 永远精确。

### 11.3 实施任务

- [ ] 为 process、cluster、AI、push 分别定义稳定排序键和批次大小。
- [ ] 使用 `findMany({ where, orderBy, take })` 循环替换“全量 IDs + slice”。
- [ ] 对需要并发处理的阶段，仅在一个数据库批次内部做有限并发。
- [ ] 每批完成后重新查询 eligible 条件，确保运行中新增且符合条件的数据是否应被本 Job 排空符合项目既定语义。
- [ ] 防止失败记录因状态未更新而在同一 Job 中立即无限重取。
- [ ] push 先按 Event 业务资格分批读取，再批量加载当前启用目标状态。
- [ ] 保持取消信号、Job 进度和阶段完成语义。
- [ ] 更新 README 中“批次是 chunk，不是完成边界”的实现说明。

### 11.4 验收标准

- [ ] 任一阶段不再先加载全部待处理 ID。
- [ ] 单次查询和单批内存对象数量有明确上限。
- [ ] 积压超过一个批次时，Job 会持续处理到当前 eligible 集合为空。
- [ ] 重试退避、取消和全局 AI Provider 故障不会造成死循环。

## 12. 工作流七：聚类候选召回收敛（P1）

### 12.1 当前问题

- `src/lib/event-clustering-service.ts` 当前可能先调用 AI 生成 EventKey。
- 随后读取最近 7 天全部 active Event，并为每个 Event 读取最多 8 篇成员，在 Node 中计算标题相似度。
- 即使最终只保留少量候选，数据库读取和字符串计算仍随近期 Event 数量增长。
- AI 应用于真正歧义判断，而不是默认参与每篇文章的候选键生成。

### 12.2 目标状态

候选决策顺序固定为：

1. 内容指纹、规范化标题等确定性精确匹配。
2. 使用已有 `eventKey`、规则事件键、时间窗口和必要索引进行数据库预筛。
3. 只对有限候选计算 2-4 gram/现有 overlap 相似度。
4. 规则证据仍歧义时才调用 AI 判断同一事件。
5. AI 失败或证据不足时 fail-closed：新建 `needs_review` Event，不允许公开或推送。

### 12.3 实施任务

- [ ] 先评估现有字段能否支持预筛；优先复用 `contentHash`、`eventKey`、规范化标题规则键。
- [ ] 如现有 Article 成员查询无法高效命中，可增加最小必要索引；不要新增向量字段。
- [ ] 将 `generateEventKey()` 从默认第一步改为规则键不足或候选歧义时的辅助步骤。
- [ ] 数据库预筛必须限制候选 Event 数和成员数。
- [ ] 保持精确匹配、规则匹配、AI 判断和人工修正的审计证据。
- [ ] 校准阈值时以误合并风险优先，宁可进入 `needs_review`，不要自动错误合并。
- [ ] 更新 README 中聚类顺序和 AI 使用边界。

### 12.4 验收标准

- [ ] 普通新文章不需要默认先调用 AI 生成 EventKey。
- [ ] 相似度计算只针对有限候选，而不是最近 7 天全量 Event。
- [ ] 指纹或规范化标题精确匹配仍能直接归入。
- [ ] 歧义和 AI 故障继续 fail-closed，`needs_review` 不可公开和推送。
- [ ] 不引入 Embedding 或外部检索基础设施。
