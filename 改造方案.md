## 二、为什么必须改造

### 1. 用户反馈

项目实际运行后，最明显的问题是：

> 同一件行业事件被不同媒体换标题、换写法后，仍会作为多篇独立内容重复推送。

这会直接降低飞书群内资讯的可信度和阅读体验，也是当前最高优先级问题。

### 2. 当前机制判断错了问题类型

当前系统主要在判断：

> 两篇文章的正文、数字或关键点是否足够相似。

真正需要判断的是：

> 两篇文章是否在报道同一个主体、同一个具体动作、同一件事项。

前者是“文章相似度/转载去重”，后者才是“事件聚类”。

### 3. 已确认的事件边界

只有满足以下条件，才应归入同一事件：

> 同一主体 + 同一具体动作 + 同一事项。

示例：

- “某品牌宣布融资”与“该轮融资正式完成”：可以视为同一事件的不同阶段。
- “某品牌完成融资”与“某品牌新开门店”：是两个事件。
- 仅仅品牌相同、行业相同、话题相似：不能合并。

第一版不自动处理“事件后续进展”，详见后文。

## 三、已经确认的产品决策

后续实施不得重新发散以下结论。

### 1. 唯一核心链路

    数据源采集
        ↓
    Article 原始文章素材
        ↓
    实时事件聚类
        ↓
    Event 行业事件
        ↓
    事件级 AI 分析
        ↓
    事件级人工审核
        ↓
    网站公开 / 飞书推送

### 2. 网站和推送的正式内容单位

- Article 只是来源素材。
- Event 才是业务内容。
- 首页一件事件只显示一张卡片。
- 详情页展示事件内容及其多个报道来源。
- 飞书推送 Event，不直接推送 Article。
- 后台日常审核 Event，不逐篇审核同一事件下的 Article。

### 3. 实时性

- 不设置 1～2 小时观察期。
- 第一篇形成新事件后，完成分析并达到质量门槛即可推送。
- 后续归入同一事件的文章只补充来源，不重复推送。
- “实时”指不增加人为等待；仍需经过正常抓取、聚类、分析和推送流水线，允许秒级处理时间。

### 4. 聚类判断策略

- Embedding 负责语义候选召回。
- 标题字符级 2～4 gram + Overlap Coefficient 负责二次确认。
- 少量模糊案例才调用 AI 判断是否为同一事件。
- 算法和 AI 均无法确定时，优先创建新事件，避免错误合并。
- AI 调用发生技术故障时，不得当成“无法确定”；文章进入待聚类重试，不能绕过聚类直接推送。

### 5. 时间边界

- 7 天内：参与自动事件聚类。
- 8～15 天：只记录“疑似相关事件”，不自动合并，不影响推送。
- 16～30 天：只记录“历史相关事件”，不自动合并，不影响推送。
- 超过 30 天：不再自动关联。

### 6. 第一版不做事件进展自动判断

第一版规则保持简单：

- 新事件：正常分析、审核、公开、推送。
- 同一事件后续文章：只补充来源，不重新推送。
- 确有重大进展：管理员手动拆分为新事件，再决定是否重新分析和参与推送。

第一版不自动判断“是否出现实质性新事实”，避免把最难、最主观的尺度带入 P0。

### 7. 人工纠错

聚类纠错只提供两个核心动作：

- 合并事件。
- 拆分文章。

每次操作都保存为审计和评估样本，但第一版不自动训练、不自动调整阈值、不自动修改规则。

### 8. 技术边界

- 继续使用 Next.js 单体、单进程、Prisma 和 SQLite。
- 向量直接保存在 SQLite。
- 仅比较有限时间窗口内的 Event，不引入 PostgreSQL、pgvector、Milvus、ElasticSearch 或独立向量数据库。
- Embedding 必须封装为可替换 Provider。
- 不新增独立队列、微服务或多实例协调系统。

这套机制适合识别转载或正文高度重叠，但不能稳定表示“一个事件包含多篇不同写法的报道”。

### 3. 当前 Article 承担了过多职责

prisma/schema.prisma 中，Article 同时保存：

- 原始抓取内容。
- AI 摘要、品牌、分类、标签和关键点。

- 浏览量和原文点击量。

这导致 Article 同时是：

> 抓取素材 + AI 分析对象 + 审核对象 + 发布对象 + 推送对象。

这是当前代码边界难以继续维护的根本原因。

### 5. 当前机制与新原则直接冲突的位置

| 当前行为 | 新原则 |
|---|---|
| process 阶段文章级去重失败后记录错误并继续 | 聚类服务技术失败必须阻断该文章后续分析和推送 |
| AI 对每篇 Article 分析 | AI 对每个新 Event 分析 |
| 重复文章被标记 skipped | 同事件 Article 正常保留并关联到同一 Event |
| Article 保存审核、公开、推送状态 | 这些状态只允许存在于 Event |
| public API 直接读取 Article | public API 只读取 Event |
| 飞书卡片直接基于 Article | 飞书卡片基于 Event 和代表 Article |
| 后台逐篇人工归类 | 后台按 Event 归类，Article 只作为来源证据 |

### 6. 应继续复用的现有能力

本次不是推翻全部项目，下列能力应保留：

- src/lib/execution.ts 的单 Job 编排与全局互斥。
- Job 进度、停止、心跳和抓取记录。
- Source、FetchLog、DiscardedItem、Keyword、KeywordCandidate。
- 关键词门控和未命中候选词闭环。
- AI Provider、重试、熔断和结构化输出基础能力。
- 评分策略、人工校准思路和设置目录。
- 公开快照思路。
- 飞书多 Webhook 投递、失败重试和脱敏日志。
- 后台 Token 会话、限流和公开 API 边界。
- 当前公开端视觉风格；本轮不做 UI 重设计。

## 五、目标领域模型

### 1. Article：原始素材

Article 只负责回答：

> 这篇原始报道来自哪里，写了什么，何时发布，被归到了哪个事件？

建议保留的字段类别：

- 标识：id、sourceId、url。
- 原始信息：title、originalSource、rawContent、cleanContent、articleBody、contentHash。
- 抓取状态：fetchStatus。
- 时间：publishedAt、createdAt、updatedAt。
- 事件关系：eventId、clusterStatus、clusteredAt。
- 聚类重试：clusterRetryCount、nextClusterRetryAt、clusterError。
- 向量：embedding、embeddingModel、embeddingDimensions、embeddedAt。

建议新增状态：

- clusterStatus=pending：等待聚类。
- clusterStatus=clustered：已进入 Event。
- clusterStatus=failed：聚类发生技术错误，等待重试。
- clusterStatus=needs_review：算法有效完成，但结果为保守新建事件并提示复核。

建议从 Article 删除的字段类别：

- AI 分析和评分字段。
- AI 快照及人工覆盖字段。
- reviewStatus、reviewReasonTags、reviewedAt。
- publicOverride、publicStatus、publicPublishedAt 等公开字段。
- duplicateOfId、duplicateStatus、dedupOverride、dedupDetail 等旧重复状态。
- viewCount、originalClickCount。
- pushedAt、nextRetryAt。

核心约束：

- Article 不能直接公开。
- Article 不能直接推送。
- Article 不能成为日常人工审核对象。
- 同一事件的不同 URL、不同媒体报道全部保留为多个 Article。

### 2. Event：唯一业务内容

Event 负责回答：

> 这是什么行业事件，系统如何理解它，是否公开，是否推送，人工如何审核？

建议字段类别：

#### 事件身份

- id
- status：active / merged
- mergedIntoId：事件合并后的规范 Event。
- representativeArticleId：当前代表文章。
- firstSeenAt
- lastSeenAt
- articleCount

#### 聚类表示

- centroidEmbedding
- embeddingModel
- embeddingDimensions
- clusterVersion

#### 事件内容与 AI

将当前 Article 上的以下能力迁入 Event：

- title
- summary
- brand
- category
- tags
- keyPoints
- relevance
- eventScore
- contentScore
- rawScore
- score
- adProbability
- aiConfidence
- isAd
- aiStatus
- aiRetryCount
- nextAiRetryAt
- aiModel
- aiProvider
- promptHash
- promptVersion
- scorePolicyVersion
- scorePolicySnapshot
- aiSnapshot
- manualOverrides
- manualCorrectedAt

#### 审核与公开

- reviewStatus
- reviewReasonTags
- reviewedAt
- publicOverride
- pinUntil
- publicStatus
- publicPublishedAt
- publicRevokedAt
- publicPublicationReason
- publicPublicationEvaluatedAt
- publicContentUpdatedAt

#### 推送与指标

- pushedAt
- nextPushRetryAt
- viewCount
- originalClickCount

#### 时间

- createdAt
- updatedAt

核心约束：

- 只有 status=active 的 Event 能进入公开列表和推送队列。
- Event 的公开、审核、评分和推送状态只能维护一份。
- 代表 Article 只是事件的主要来源，不重新拥有业务状态。
- Event 被合并后，旧事件不得继续公开或推送。

### 3. EventClusterAudit：聚类与纠错证据

需要新增不可变审计记录，用于解释和评估，不用于自动学习。

建议字段：

- id
- articleId
- assignedEventId
- candidateEventId
- actor：system / admin
- action：create / attach / fallback_create / merge / split
- semanticSimilarity
- titleOverlap
- contentFingerprintMatched
- aiUsed
- aiDecision
- aiProvider
- aiModel
- clusterVersion
- evidence：JSON，保存候选、阈值、理由和摘要证据。
- createdAt

用途：

- 后台展示“为什么归到这个事件”。
- 支持人工合并、拆分后的追溯。
- 后续离线评估阈值。
- 不得据此自动修改生产规则。

### 4. EventRelation：非阻断的历史关联

用于承载 8～30 天规则，不参与自动聚类、公开或推送。

建议字段：

- id
- fromEventId
- toEventId
- relationType：possible_related / historical_related
- semanticSimilarity
- titleOverlap
- evidence
- createdAt

规则：

- 8～15 天：possible_related。
- 16～30 天：historical_related。
- 同一对 Event 保持唯一关系。
- 该表只是详情提示，不能改变 pushedAt、publicStatus 或 eventId。

### 5. PushLog

- 将 articleId 改为 eventId。
- 飞书卡片使用 Event 内容。
- 原文链接来自 Event.representativeArticleId。
- 合并事件不会补发或撤回历史飞书消息。

### 6. 向量存储

建议使用 SQLite BLOB：

- Prisma 字段类型使用 Bytes。
- 保存 L2 归一化后的 Float32 向量。
- 使用统一 encode/decode helper，禁止业务代码各自解析。
- Article 保存自身向量。
- Event 保存成员向量的归一化中心向量。
- 同时保存模型名和维度，禁止不同模型的向量混合比较。

若实施时选用 JSON 字符串，必须说明理由并集中封装；不得把向量解析散落在 Route Handler 或组件中。

## 六、实时事件聚类算法

### 1. 总体流程

    Article 完成正文抓取和关键词门控
        ↓
    生成并缓存 Article Embedding
        ↓
    查询最近 7 天 active Event 的中心向量
        ↓
    余弦相似度召回 Top-K 候选
        ↓
    标题 2～4 gram + Overlap Coefficient 二次确认
        ↓
    明确相同：加入 Event
    明确不同：创建 Event
    模糊区间：AI 裁决
        ↓
    更新 Event 中心向量、来源数、时间和代表文章
        ↓
    新 Event 进入 AI 分析；已有 Event 不重复分析和推送

### 2. Embedding Provider

新增独立边界，例如：

- src/lib/embedding/provider.ts
- src/lib/embedding/openai-compatible.ts
- src/lib/embedding/vector-codec.ts

建议接口职责：

- 输入一段标准化文本。
- 输出模型名、维度和 L2 归一化向量。
- 统一处理超时、重试、鉴权和技术错误。

第一版建议实现 OpenAI-compatible /embeddings 适配器，并新增独立配置：

- embedding_base_url
- embedding_api_key
- embedding_model
- embedding_dimensions

注意：

- 不得默认把当前聊天模型当作 Embedding 模型。
- Embedding 配置缺失或服务技术失败时，聚类阶段失败并重试。
- Embedding Provider 与现有 Chat Provider 分开，避免模型能力混淆。
- 后续可增加本地模型适配器，但不属于本轮必做。

建议向量输入由以下内容组成：

- 标题。
- 来源名称。
- 正文前一段有限长度的清洗文本。

不得把完整超长正文全部发送给 Embedding 服务；输入规则应集中在一个函数中并带版本号。

### 3. 标题 n-gram

新增纯函数模块，例如 src/lib/event-clustering/title-overlap.ts。

标准化步骤：

1. Unicode NFKC 标准化。
2. 英文字母转小写。
3. 去除空白和常见标点。
4. 保留中文、英文、数字和有意义的专名字符。
5. 生成字符级 2-gram、3-gram、4-gram 集合。

相似度：

    overlap(A, B) = |A ∩ B| / min(|A|, |B|)

比较范围：

- 新 Article 标题。
- 候选 Event 的代表 Article 标题。
- 候选 Event 最近若干篇成员 Article 标题。

采用最大标题覆盖度作为该候选的 lexical 证据，避免只比较代表标题造成漏判。

### 4. 决策分层

具体阈值必须集中在一个契约文件中，例如 src/contracts/event-clustering.ts，不允许散落魔法数字。

第一版采用保守策略：

#### 强确定相同

满足任一硬条件，或同时满足高语义相似和高标题覆盖：

- 7 天内内容指纹完全相同。
- 7 天内标准化标题完全相同。
- Embedding 高相似 + n-gram 高覆盖。

结果：直接关联已有 Event。

#### 强确定不同

- 7 天内没有达到候选召回下限的 Event。
- 或候选在语义和标题证据上均明显不足。

结果：创建新 Event。

#### 模糊区间

常见情况：

- 语义很近，但标题写法差异大。
- 标题包含相同品牌和动作词，但语义证据不足。
- 多个 Event 得分接近。

结果：只对少量 Top 候选调用 AI 裁决。

阈值不能在尚未固定 Embedding 模型前凭空确定。实施时应：

1. 先固定模型和输入版本。
2. 使用保守初始阈值。
3. 用实际新抓数据和人工合并/拆分样本调整。
4. 优先保证错误合并率低于 1%，不要为了追求“零重复”降低合并门槛。

### 5. AI 模糊裁决

AI 只回答：

> 新文章与候选事件是否属于同一主体、同一具体动作、同一事项？

输入应限制为：

- 新文章标题和必要正文片段。
- 候选事件标题。
- 候选事件代表文章摘要片段。
- 候选事件最近成员标题。
- 两者时间。

输出使用严格 JSON：

- sameEvent：true / false
- confidence：0～100
- subject
- action
- matter
- reason

处理规则：

- 有效返回且高置信 sameEvent=true：加入候选 Event。
- 有效返回 sameEvent=false：创建新 Event。
- 有效返回但置信不足或明确 uncertain：创建新 Event，并将 clusterStatus 标记为 needs_review。
- API 超时、鉴权失败、格式持续无效等技术故障：clusterStatus=failed，按退避策略重试；不得创建 Event 后继续推送。

### 6. 同批次并发保护

当前项目单进程且一个批量 Job 全局互斥，可利用这一事实保持简单：

- Embedding 生成可以有限并发。
- 聚类归属写入必须按有效发布时间排序后串行执行。
- 每次归属在事务内完成。
- 新建 Event 后，后续同批 Article 必须立即能召回该 Event。

这样可以避免两篇同时到达的同事件文章各自创建一个 Event。

### 7. Event 中心向量

- 新 Event 的中心向量等于首篇 Article 向量。
- 新 Article 加入后，按成员数更新平均向量并重新 L2 归一化。
- 合并或拆分后，从实际成员 Article 重新计算，不使用增量猜测。
- Event.articleCount、firstSeenAt、lastSeenAt 与中心向量在同一事务中更新。

### 8. 7/15/30 天处理

- 自动归属只查询最近 7 天 Event。
- 新建 Event 后，可用同一向量从 8～30 天范围查找少量相关 Event。
- 8～15 天写 possible_related。
- 16～30 天写 historical_related。
- 这一步失败不能阻塞新 Event 的主流程，因为它不是去重门槛，只是辅助展示。

## 七、代表文章规则

第一版不使用 LLM 选择代表文章，采用可解释的确定性规则。

优先级：

1. 管理员手动指定的代表文章。
2. 更像直接来源、不是明显转载的文章。
3. 正文更完整、有效文本更多的文章。
4. 发布时间更精确的文章。
5. 其他条件相同时，较早发布的文章优先。

建议集中为 src/lib/event-representative-service.ts，输出选择理由。

规则：

- 新成员进入 Event 后允许自动更换代表 Article。
- 更换代表 Article 不产生新的飞书推送。
- 更换代表 Article 不自动触发 Event AI 重分析。
- Event 标题只有在未被人工覆盖时，才允许跟随代表 Article 更新。
- 管理员应能在事件来源列表中手动指定代表文章。

第一版不尝试自动判断哪篇文章包含“实质新增事实”，避免重新引入事件进展复杂度。

## 八、目标流水线

### 1. 新主链路

    collect → process → cluster → analyze-events → push-events

### 2. collect

继续复用现有采集逻辑：

- Source 和 FetchLog 保持。
- 相同 URL 仍由唯一约束阻止重复创建。
- 新 Article 初始 clusterStatus=pending。

### 3. process

保留：

- 详情抓取。
- 正文清洗。
- 发布时间修复。
- 关键词门控。
- 未命中候选词和 DiscardedItem。

删除或迁出：

- findDuplicateArticle 对 Article 的 skipped 写入。
- duplicateStatus、duplicateOfId、dedupOverride。
- process 内 Article 公开快照刷新。
- “去重失败后继续走 AI”的容错方式。

内容指纹只作为后续 Event 聚类的强证据，不再把 Article 直接判死。

### 4. cluster

新增 src/lib/pipeline/cluster.ts：

- 查询 fetchStatus=fetched、clusterStatus=pending/failed 且到达重试时间的 Article。
- 生成或复用 Embedding。
- 调用 EventClusteringService。
- 新建或关联 Event。
- 写 EventClusterAudit。
- 聚类技术失败时保留 Article，不进入 Event 分析和推送。
- 向 Job 写入 cluster 阶段进度。

需要扩展：

- JobStage 增加 cluster。
- JobType 可增加 cluster，供管理员单独重试。
- full Job 在 process 后、analyze 前执行 cluster。

### 5. analyze-events

将当前 Article AI 分析迁为 Event AI 分析：

- 只查询 Event.aiStatus=pending/failed。
- 每个新 Event 只做一次主分析。
- 输入以代表 Article 正文为主，可附带同 Event 其他来源标题。
- AI 输出和评分策略写入 Event。
- 新 Article 加入已有 Event 时，第一版不自动重分析。
- 管理员可以手动“重新分析事件”。

当前人工校准、AI 快照和评分重算能力如在最终基线仍存在，应迁移到 Event，不能继续留在 Article。

### 6. publish-events

将公开规则迁入 Event：

- 评分、相关度、软文、人工归类、人工覆盖均读取 Event。
- publicStatus 只存在于 Event。
- 公开快照服务改为 EventPublicationService。
- Article 内容变化只能通过更新所属 Event 的代表来源或人工重分析影响公开内容。

### 7. push-events

将推送策略迁入 Event：

- pushableWhere 查询 Event。
- Event.pushedAt 是“该事件是否已推送”的唯一事实源。
- 同 Event 新增 Article 不清空 pushedAt。
- PushLog 关联 Event。
- 飞书卡片展示 Event 标题、摘要、评分、标签及代表来源链接。
- 聚类失败的 Article 不可能进入推送查询。

## 十、目标 API 边界

### 1. 后台 Event API

建议：

- GET /api/events：事件列表。
- GET /api/events/[id]：事件详情和来源摘要。
- PATCH /api/events/[id]：人工修正事件内容、公开策略和代表文章。
- POST /api/events/review：单个或批量事件归类。
- POST /api/events/reprocess：重新分析 Event。
- POST /api/events/merge：合并事件。
- POST /api/events/[id]/split：从事件中拆分指定 Article。
- GET /api/events/[id]/articles：查看事件下全部来源。

### 2. Article API

Article API 只承担：

- 查看原始素材。
- 查看抓取结果和正文。
- 查看聚类状态与证据。
- 对失败素材执行重新抓取或重新聚类。

不得再承担：

- 人工归类。
- 公开覆盖。
- 置顶。
- AI 评分校准。
- 单篇推送。

### 3. 公开 API

建议：

- GET /api/public/events
- GET /api/public/events/[id]
- POST /api/public/events/[id]/click

公开 DTO 不得暴露：

- Embedding。
- 聚类阈值。
- AI 裁决细节。
- 内部审核日志。
- Webhook 或推送日志。

### 4. 公开页面路径

为了减少 SEO 和页面结构变动，可继续保留：

- /
- /news/[id]

但 /news/[id] 的 id 必须是 Event.id，页面内容必须来自 Event。

旧 Article URL 不需要兼容，因为旧业务数据明确不迁移。

## 十一、后台交互

### 1. 情报收件箱

默认列表从“全部文章”改为“全部事件”。

建议显示：

- Event 标题。
- 代表来源。
- 来源数量。
- 首次/最近发现时间。
- AI 状态和评分。
- 人工归类。
- 实际公开状态。
- 实际推送状态。
- 是否需要聚类复核。

右侧详情显示：

- Event 摘要、关键点、品牌、分类、标签和评分。
- 代表文章。
- 全部来源列表。
- 每篇 Article 的来源、标题、发布时间和原文链接。
- 聚类证据摘要。
- 可能相关/历史相关事件。

### 2. 合并事件

管理员选择两个 Event，指定保留的目标 Event。

服务必须在事务中：

1. 把来源 Article 移到目标 Event。
2. 重新计算目标 Event 中心向量、时间、文章数和代表文章。
3. 把来源 Event 标记为 merged，并写 mergedIntoId。
4. 让来源 Event 从公开列表和推送队列消失。
5. 保存 EventClusterAudit。
6. 不补发飞书消息。

如果来源 Event 已经有公开 URL，详情访问应解析到规范 Event，而不是返回两份内容。

### 3. 拆分文章

管理员在 Event 来源列表中选择一篇或多篇 Article，创建新 Event。

服务必须：

1. 从原 Event 移出所选 Article。
2. 用所选 Article 创建新 Event。
3. 分别重算两个 Event 的中心向量、文章数、时间和代表文章。
4. 保存 EventClusterAudit。
5. 在确认框中提供“拆分后重新分析并按新事件参与推送”的明确选项，默认关闭，避免纠错动作意外补推旧内容。

### 4. 不暴露复杂参数

第一版后台不增加：

- Embedding 阈值输入框。
- n-gram 阈值输入框。
- 自动学习开关。
- 事件进展判断规则编辑器。

技术阈值集中在契约与服务中，通过审计样本受控调整。

这是结构性改造，后续实施时必须补充针对性测试。

### 1. 聚类算法

- [ ] 标题标准化。
- [ ] 2/3/4-gram 生成。
- [ ] Overlap Coefficient 长短标题场景。
- [ ] 向量编解码、归一化和余弦相似度。
- [ ] 7 天自动聚类边界。
- [ ] 8～15 天疑似相关。
- [ ] 16～30 天历史相关。
- [ ] 超过 30 天不关联。
- [ ] 高置信相同、高置信不同和模糊 AI 三条路径。
- [ ] AI 技术失败进入重试，不创建可推送 Event。

### 2. 数据一致性

- [ ] Article 只能属于一个 active Event。
- [ ] Event 代表 Article 必须属于该 Event。
- [ ] 合并后来源 Event 不再公开和推送。
- [ ] 拆分后两个 Event 的向量、时间和文章数正确。
- [ ] 同批两篇相同事件文章不会并发创建两个 Event。

### 3. 推送和公开

- [ ] 新 Event 达到门槛后只推送一次。
- [ ] 已推送 Event 新增 Article 不再次推送。
- [ ] 不同 Event 不被错误阻止推送。
- [ ] public API 只返回 active 且 published 的 Event。
- [ ] Article 无法绕过 Event 直接公开或推送。
- [ ] 浏览和点击不改变公开内容更新时间。

### 4. 人工操作

- [ ] 合并事件不会产生新推送。
- [ ] 拆分默认不补推。
- [ ] 拆分时明确选择参与推送后，按新 Event 正常走分析和门槛。
- [ ] 人工代表文章优先于自动选择。
- [ ] 人工校准只修改 Event。

### 5. 交付验证

完成代码后按风险执行：

- Prisma Client 生成。
- migration 状态。
- TypeScript 类型检查。
- ESLint。
- 聚类和事件流程的针对性 Vitest。
- 默认测试集。
- 生产构建。
- git diff --check。

## 十六、业务验收标准

第一版不追求完美聚类，达到以下条件即可投入实际运行：

- 同一事件重复推送减少 80% 以上。
- 不同事件错误合并低于 1%。
- 正常聚类保持秒级，不增加人工观察时间。
- 模糊案例优先新建 Event，而不是冒险合并。
- 聚类服务技术故障时不会绕过去重直接推送。
- 网站、后台和飞书全部以 Event 为正式内容单位。
- Article 仅作为原始来源存在。

建议利用 EventClusterAudit 对实际新数据定期抽样：

- 查看被自动合并的 Event。
- 查看 needs_review 新事件。
- 记录人工合并与拆分次数。
- 用这些样本调整阈值，但不自动学习。

## 十七、本轮明确冻结

在新链路稳定前，不开发：

- UI 美化和大规模视觉改版。
- 新工具和数据页面。
- 新数据源扩张。
- 个性化推荐。
- 用户画像。
- 自动学习和在线训练。
- FTRL、MMR 等排序系统。
- 自动事件进展识别。
- 同一事件自动二次推送。
- 每日/每周事件简报。
- PostgreSQL 或向量数据库。
- 独立队列、微服务和多实例。
- 旧业务数据迁移与兼容。

所有新增需求都应先回答：

> 它是否直接改善事件聚类、重复推送、事件审核或事件发布？

不能直接改善上述 P0 的功能，暂不进入本轮。

## 十八、给后续实施 AI 的最终约束

1. 先重新核对代码，不要把本文的当前状态当成永远不变的事实。
2. 不得覆盖用户在本文之后完成的代码。
3. 不保留 Article/Event 双轨业务逻辑。
4. 不为旧业务数据写兼容分支。
5. 不把 Embedding、n-gram、AI 裁决各自做成互不相干的三套事实。
6. 所有聚类阈值、版本和输入规则必须集中管理。
7. Route Handler 只做参数适配，聚类、合并、拆分、公开和推送规则必须位于 Service。
8. Article 不得重新获得审核、公开或推送职责。
9. 聚类技术失败必须 fail closed；有效但不确定才允许保守创建新 Event。
10. 每完成一个阶段，同步更新本文复选框和相关文档。

最终完成标志不是“新增了 Event 表”，而是：

> 从采集、分析、审核、公开到推送，系统只有一条 Article → Event 的业务链路，并且不存在任何 Article 直接对外发布的旁路。
