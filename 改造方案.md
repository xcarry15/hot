# Hot2-S 情报收件箱与事件去重改造方案

> 实施状态：已完成（2026-07-18）。本文同时作为当前业务边界与回归验收基线。

## 1. 改造目标

本轮改造只解决三个核心问题：

1. 情报收件箱保持“全量文章表格＋人工改分台”，不再依赖不准确的“待处理”结果作为主要入口。
2. 同一新闻事件被多个来源重复报道时，只公开、只推送一次，同时保留全部原始文章供人工查看和校准。
3. 人工修改能够沉淀为可复查样本，用于后续调整 AI 提示词和聚类规则，但不建设自动训练系统。

项目规模预计少于 20 个数据源、每天少于 200 篇文章，因此优先采用简单、可解释、可人工纠错的单体方案，不引入向量数据库、微服务、独立队列或在线学习系统。

## 2. 产品形态

### 2.1 情报收件箱

情报收件箱继续以 Article 为操作对象：

- 左侧展示全量文章，默认不隐藏已处理、低分、重复来源或已公开文章。
- 保留搜索、来源、状态、评分和时间筛选，但筛选只缩小视图，不改变文章业务状态。
- 右侧继续展示文章详情、AI 原始结果和人工改分表单。
- 人工可修改评分、相关度、分类、标签、摘要、关键点、软文判断和公开策略。
- 人工值覆盖 AI 值；重新分析时保留人工覆盖，并允许恢复 AI 原值。
- 增加所属事件、聚类状态、同事件来源和代表文章信息。

“需关注”仅作为辅助提示，不能替代全量列表，也不能决定文章是否可见。

### 2.2 事件去重

新增轻量 Event，作为公开和推送的唯一去重门禁：

```text
Article：原始文章、AI 分析、人工校准
   ↓ 归属
Event：同一新闻事件、代表文章、公开状态、推送状态
```

同一 Event 下可以保留多篇 Article，但：

- 首页最多展示一张卡片。
- 飞书最多首发一次。
- 公开内容和推送内容取代表 Article 的最终结果。
- 后续新增同事件来源不能清空已推送状态。

## 3. 数据职责

### 3.1 Article

Article 负责：

- 来源、标题、链接、正文和发布时间。
- 抓取、处理和 AI 分析状态。
- AI 摘要、品牌、分类、标签、关键点和评分。
- AI 原始快照、人工覆盖和人工审核结果。
- 内容指纹、事件键和聚类状态。
- 所属 Event。

Article 不再负责：

- 判断是否已经对外推送。
- 以“重复文章”状态阻止后续分析。
- 绕过 Event 直接进入公开或推送链路。

### 3.2 Event

Event 只保留事件级必要状态：

- active 或 merged 状态。
- 合并后的目标 Event。
- 代表 Article。
- 是否人工指定代表 Article。
- 首次发现时间、最近发现时间和文章数量。
- 公开状态和公开时间。
- 推送成功时间和推送重试时间。

Event 不复制 Article 上的整套 AI 分析和人工改分字段，避免形成两套事实。

### 3.3 EventClusterAudit

EventClusterAudit 保存聚类和人工纠错记录：

- 哪篇 Article 被分配到哪个 Event。
- 系统创建、规则合并、AI 合并、保守新建、人工合并、人工拆分或更换代表文章。
- 候选 Event、置信度和判断证据。
- 操作者和发生时间。

审计记录用于解释结果、定位错误和形成校准样本，不直接触发自动学习。

## 4. 主业务流程

```text
collect → process → cluster → ai → publish / push
```

### 4.1 collect

- 抓取来源并创建 Article。
- URL 唯一约束继续阻止完全相同链接重复入库。
- 新 Article 初始为待聚类状态。

### 4.2 process

- 抓取正文并清洗内容。
- 修复发布时间。
- 执行关键词门控。
- 计算内容指纹。
- 内容相似只作为事件聚类证据，不再直接把 Article 判死为重复文章。

### 4.3 cluster

对通过处理阶段的 Article 执行轻量聚类：

1. 生成标准化标题和 `eventKey`；AI 提取失败时回退到规则标题键。
2. 查询最近 7 天的 active Event。
3. 使用内容指纹、标题和 eventKey 筛选少量候选。
4. 明确相同则加入已有 Event。
5. 没有可信候选则创建新 Event。
6. 证据模糊时只对最相关的少量候选调用 AI 判断。
7. AI 有效返回但无法确定时，保守创建新 Event，并标记需要人工复核。
8. AI 或聚类服务技术失败时，标记失败并重试，不得进入公开或推送。

`eventKey` 使用简单格式：

```text
主体|动作|事项
```

例如：

```text
胖东来|开店|郑州首店
瑞幸咖啡|发布财报|2026年第二季度
```

eventKey 只用于候选召回，不能单独决定合并。

### 4.4 ai

- AI 继续分析 Article，而不是 Event。
- 未成功归属 Event 的 Article 不进入 AI、公开和推送。
- AI 结果只提供初始判断，人工覆盖始终优先。

### 4.5 publish

- 公开查询以 active Event 为入口。
- 代表 Article 必须满足现有公开规则。
- 一个 Event 最多出现在首页一次。
- `/news/[id]` 使用 Event ID，代表文章变化不改变公开 URL。
- 详情页展示代表 Article，并列出同事件的其他来源。

### 4.6 push

- 推送查询以 active、未推送 Event 为入口。
- 推送资格取决于代表 Article 的最终人工校准结果。
- 推送成功后写入 Event 的推送时间。
- 同 Event 后续加入新 Article 不重复推送。
- PushLog 关联 Event，并记录发送时使用的代表 Article。

## 5. 事件判定规则

同一事件的判断原则是：

> 同一主体＋同一具体动作＋同一事项。

以下情况可以合并：

- 相同内容指纹。
- 标准化标题完全一致。
- eventKey 一致，且标题中的地点、时间、产品、批次等关键信息一致。
- AI 对模糊候选给出明确、可解释的同事件结论。

以下情况不能仅因主体相同而合并：

- 同一品牌的开店和融资。
- 同一品牌在不同城市开店。
- 同一公司不同财报周期。
- 仅行业、品牌或话题相近。

证据不足时优先新建 Event，避免错误合并造成不同新闻被永久挡住。

第一版只比较最近 7 天，不处理长期事件关系、自动进展识别和同事件二次推送。

## 6. 代表文章

新 Event 默认使用第一篇 Article 作为代表文章。后续可按以下优先级更新：

1. 管理员手动指定。
2. 人工归类为重要。
3. 最终分和相关度更高。
4. 正文更完整。
5. 发布时间更早。

必须满足：

- 代表 Article 属于当前 Event。
- 人工指定后，自动流程不得覆盖。
- 代表文章变化只影响网站后续展示，不修改或撤回已发送消息。
- 删除代表 Article 时，必须先选择新代表；Event 没有成员时才删除 Event。

## 7. 人工纠错

情报收件箱只增加三个事件级动作。

### 7.1 合并事件

- 将来源 Event 的全部 Article 移入目标 Event。
- 重算文章数、首次/最近发现时间和代表文章。
- 来源 Event 标记为 merged，不再公开和推送。
- 若任一 Event 已推送，合并后目标 Event 继续视为已推送。
- 不补发、不撤回历史消息。
- 写入审计记录。

### 7.2 拆分文章

- 将选中的 Article 移入新 Event。
- 分别重算新旧 Event。
- 拆分出的新 Event 默认保持未推送，因为它已被人工确认为独立事件，应进入正常推送资格判断。
- 写入审计记录。

### 7.3 指定代表文章

- 只能选择当前 Event 的成员 Article。
- 保存人工指定状态。
- 写入审计记录。

## 8. 人工结果如何校准 AI

本项目不做在线训练。人工结果通过“记录差异、定期复盘、人工调整”形成轻量闭环。

### 8.1 需要保留的校准信号

Article 人工改分时保留：

- AI 原始值。
- 人工最终值。
- 被修改的字段。
- 修改时间。

Event 人工纠错时保留：

- 人工合并：说明系统发生漏聚类。
- 人工拆分：说明系统发生错误合并。
- 人工更换代表文章：说明自动代表规则不合理。
- 纠错前的候选、置信度、规则版本和判断证据。

### 8.2 校准方式

每积累一批真实数据后，人工抽查以下样本：

- AI 与人工评分差异较大的 Article。
- `needs_review` Article。
- 人工合并和拆分记录。
- 重复推送漏网案例。
- 不同事件被错误合并的案例。

根据样本只做三类调整：

1. 调整 AI 提示词和示例。
2. 调整 eventKey 提取规则或聚类判断规则。
3. 调整少量集中管理的阈值。

每次调整必须使用历史人工样本做前后对比，确认漏合并减少且错误合并没有明显增加后再启用。

不允许：

- 根据单条人工修改立即改变模型或阈值。
- 在后台提供随意调阈值的输入框。
- 自动修改提示词、规则或模型配置。
- 引入自动训练、在线学习或复杂特征平台。

## 9. 必须防止的功能与逻辑 BUG

### 9.1 列表与人工编辑

- 全量列表不能被“待处理”条件隐式过滤。
- 筛选、翻页或自动下一篇后，右侧详情必须对应当前选中 Article。
- 保存失败时不能提前更新界面或跳到下一篇。
- 重新分析不能覆盖人工修改。
- 恢复 AI 原值只能清除指定字段的人工覆盖。
- 删除、合并或拆分后必须刷新列表、详情和 Event 信息，不能保留失效选择。

### 9.2 聚类

- 同批次相同事件不能因并发创建多个 Event。
- 技术失败与业务不确定必须使用不同状态。
- 抓取记录页必须展示聚类步骤，并能筛选待聚类、聚类失败和待复核文章。
- 聚类未完成或失败时，AI 与推送状态必须显示为阻塞，不能误导为待 AI。
- 聚类失败不能绕过门禁进入 AI、公开或推送。
- merged Event 不能继续成为候选、公开内容或推送对象。
- Article 最多属于一个 Event。

### 9.3 代表文章

- 代表 Article 必须属于 Event。
- 人工指定代表不能被自动重算覆盖。
- 合并、拆分、删除 Article 后必须重算代表文章。
- 代表文章变化不能改变 Event URL，也不能造成再次推送。

### 9.4 公开与推送

- Article 不能直接公开或直接推送。
- 一个 Event 在首页最多出现一次。
- 一个 Event 最多首发推送一次。
- 新增同事件来源不能清空 Event 的推送成功状态。
- 合并不能意外补推；拆分出的独立 Event 应按自身资格正常首发。
- 推送日志必须保留 Event 和发送时代表 Article。
- 修改当前代表 Article 后，历史推送日志和来源统计不得漂移。
- 未配置可用 Webhook 时批量推送应直接跳过，不得持续制造 Event 级失败日志。
- 浏览量和原文点击量不能改变公开内容更新时间。

## 10. 代码边界

业务规则放在 Service，Route Handler 只负责鉴权、参数校验和响应转换。

| 模块 | 职责 |
|---|---|
| `prisma/schema.prisma` | Article、Event、EventClusterAudit、PushLog 数据关系 |
| `src/lib/event-clustering-service.ts` | 候选召回、规则判断、AI 裁决、事件归属 |
| `src/lib/event-service.ts` | 合并、拆分、代表文章和事件重算 |
| `src/lib/pipeline/cluster.ts` | 聚类批次、失败重试和任务进度 |
| `src/lib/review-service.ts` | Article 人工校准及相关重算 |
| `src/lib/public-article-service.ts` | Event 级公开查询和详情 |
| `src/lib/public-publication-service.ts` | 根据代表 Article 刷新 Event 公开状态 |
| `src/lib/push/*` | Event 级推送资格、投递、日志和重试 |
| `src/components/intelligence-inbox.tsx` | 全量文章工作台和事件纠错入口 |

系统只保留一条对外链路：

```text
Article → Event → 公开 / 推送
```

## 11. 实施顺序

### 阶段一：建立数据边界

- 新增 Event、EventClusterAudit 和 Article 事件归属字段。
- PushLog 改为关联 Event。
- 建立正规 Prisma migration。
- 不迁移旧业务数据，改造完成后重新采集。

完成标准：Article 能稳定归属 Event，Event 能独立记录公开和推送状态。

### 阶段二：接入轻量聚类

- 实现 eventKey、标题标准化、内容指纹和 7 天候选召回。
- 实现规则判断、少量 AI 裁决、保守新建和失败重试。
- 主链路改为 `collect → process → cluster → ai → publish/push`。

完成标准：通过处理阶段的 Article 必须先归属 Event，技术失败不能绕过聚类。

### 阶段三：切换公开与推送

- 公开列表、详情和 sitemap 改为 Event 门禁。
- 推送资格、PushLog、失败重试和飞书卡片改为 Event 门禁。
- 删除 Article 直接公开和直接推送旁路。

完成标准：同一 Event 只公开一次、只首发推送一次。

### 阶段四：完善人工工作台

- 保持全量 Article 列表和现有人工改分能力。
- 增加聚类状态、同事件来源和代表文章信息。
- 实现合并、拆分和指定代表文章。
- 保存人工校准与聚类审计记录。

完成标准：管理员可以在一个页面完成文章校准和事件纠错。

### 阶段五：清理旧实现并启用

- 删除旧 duplicate 状态、恢复 API、无引用 helper、组件和测试。
- 删除 Article/Event 双轨公开和推送逻辑。
- 同步 README 和 AGENTS。
- 清空旧业务数据并重新抓取。
- 暂停实时推送，人工抽查首轮 Event 后再恢复。

完成标准：仓库和运行时只有 Article → Event → 公开/推送一条业务链路。

## 12. 验收标准

### 12.1 情报收件箱

- 默认展示全部 Article。
- 搜索、筛选、翻页、选择和自动下一篇状态一致。
- 人工修改、恢复 AI 原值和重新分析互不冲突。
- 合并、拆分和指定代表文章后界面数据正确刷新。

### 12.2 聚类

- 相同内容指纹或相同标准化标题进入同一 Event。
- 同品牌不同动作、地点、事项或周期不会错误合并。
- 模糊案例调用 AI，AI 无法确定时保守新建。
- 技术失败进入重试，不能公开或推送。
- 合并、拆分、删除后 Event 数量、时间和代表文章一致。

### 12.3 公开与推送

- 一个 Event 在首页最多展示一次。
- 一个 Event 最多首发推送一次。
- 新增同事件 Article 不重复公开或推送。
- merged Event 和未聚类 Article 无法绕过门禁。
- 代表文章变化后 Event URL 和已推送状态保持不变。

### 12.4 人工校准闭环

- Article 人工值与 AI 原值可以比较。
- Event 合并、拆分和代表文章修改均可追溯。
- 可以从审计记录提取漏聚类和错误合并样本。
- 提示词或规则调整前后能够使用同一批人工样本比较结果。

## 13. 本轮不做

- Embedding 和向量数据库。
- EventRelation 和长期事件图谱。
- 自动事件进展识别和同事件二次推送。
- 自动训练、在线学习和自动修改规则。
- PostgreSQL、微服务和独立队列。
- 大规模 UI 改版、新工具页和数据页。
- 旧业务数据迁移与兼容。

新增需求只有在能够直接改善以下目标时才进入本轮：

- 减少重复公开或重复推送。
- 提高人工审核效率。
- 修复聚类错误。
- 降低情报收件箱的功能和逻辑 BUG。

## 14. 最终完成标准

```text
全量 Article 采集与人工校准
          ↓
Event 统一事件归属
          ↓
Event 控制唯一公开与唯一推送
```

最终必须同时满足：

- 情报收件箱仍是全量文章人工校准工作台。
- 同一事件的全部来源文章均被保留并可查看。
- 同一 Event 不会因新增来源重复公开或推送。
- 管理员能够修复错误合并、漏合并和代表文章错误。
- 人工结果能够沉淀为后续调整 AI 和规则的评估样本。
- Article 不存在绕过 Event 直接对外发布的旁路。
- 项目继续保持 Next.js 单体、Prisma、SQLite 和简单任务编排。
