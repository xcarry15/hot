Hot2 项目全面改建任务书

审查基准：xcarry15/hot，master@23e719f

一、总体结论

当前项目已经形成了比较完整的产品闭环：

数据采集 → 正文处理 → AI 分析 → 事件聚类 → 人工校准 → 公开发布 → 飞书推送

主链路和领域概念已经比较清楚，Event 作为公开与推送的去重单位也是正确方向。

但项目当前最严重的问题，不是某个公式阈值不够准确，而是以下四个基础问题：

1\.任务、锁和外部投递缺少持久化语义。

2\.聚类证据会把不同成员文章的最优特征错误拼接。

3.Event 派生状态缺少数据库级不变量和可靠重建机制。

4\.项目没有以真实标注数据衡量聚类准确率。

如果继续直接调整提示词、相似度阈值和召回数量，短期可能改善个别案例，但系统仍会持续出现：

进程重启导致任务丢失；

同一任务或同一 Event 被重复执行；

不同事件被错误合并；

Event 显示已推送，但目标内容实际上没有被投递；

Event 成员数、代表文章、公开状态和推送状态互相矛盾；

每次调整规则都不知道整体准确率到底提升还是下降。

改建时应遵循以下第一性原则：

原则一：事实必须持久化

凡是影响业务结果的状态，不能只存在于 Node.js 内存中。

包括：

谁正在执行任务；

哪个任务已经被领取；

是否要求取消；

某个 Event 是否正在推送；

某个目标是否已经投递；

某个 Event 是否需要重新计算。

原则二：一次判断的证据必须来自同一对象

不能把：

成员 A 的标题相似度；

成员 B 的正文相似度；

成员 C 的事件身份相似度；

拼成一个虚构的“最强候选”。

原则三：外部副作用无法和数据库事务原子提交

飞书已经收到消息，但数据库写日志前进程崩溃，是必然需要处理的情况。

系统必须明确采用：

至少一次；

至多一次；

或“状态未知后人工确认”；

不能假设数据库日志存在就等于发送成功，也不能假设日志不存在就可以安全重发。

原则四：Event 的公开、代表、计数和推送状态都是派生状态

这些状态必须：

能从 Article、EventMember、Delivery 等基础事实重新计算；

有一致性检查；

有失败后的自动修复机制。

原则五：聚类效果必须可测量

没有真实标注集，就无法判断：

阈值调整是否真的有效；

提高召回率是否同时造成大量误合并；

提示词变化是否只是改变了输出形式；

AI 调用成本是否值得。



二、严重问题优先级

P0-1：任务执行和全局写锁只存在于进程内存

当前 runJob() 的行为是：

1\.在内存中申请全局锁；

2\.写入一条 running Job；

3\.使用 void runPipeline(...) 启动脱离请求生命周期的异步任务；

4.API 立即返回。

项目明确使用模块级内存变量作为单任务门禁。

mutation-guard.ts 中的 activeMutation 也只是当前进程中的变量，并明确依赖“只运行一个进程”的假设。

直接后果

1\. 进程重启后任务消失

数据库中可能仍然显示 running，但已经没有任何代码继续执行。

2\. 多实例时锁完全失效

两个 PM2 实例、滚动发布期间的新旧实例，或者未来拆分 Worker 后，都可以同时启动任务。

3\. 启动清理可能误杀其他实例的任务

当前启动时会把数据库里所有 running Job 直接改成失败。

如果存在两个实例，实例 B 启动时可能把实例 A 正在执行的任务标记为失败。

4\. 调度器也只在进程内防重

调度器使用 globalThis.\_\_newsSchedulerStarted，它只能保证单个进程内不重复启动，不能保证多个进程只调度一次。

必须改建为持久化 Job Lease

Job 模型建议增加

status:

&#x20; queued

&#x20; running

&#x20; cancel\_requested

&#x20; succeeded

&#x20; failed

&#x20; cancelled



leaseOwner

leaseExpiresAt

heartbeatAt

attempt

maxAttempts

idempotencyKey

availableAt

startedAt

completedAt

lastError

新的执行流程

API / Scheduler

&#x20;   ↓

只创建 queued Job

&#x20;   ↓

Worker 原子领取 Job

&#x20;   ↓

写入 leaseOwner + leaseExpiresAt

&#x20;   ↓

周期性续租

&#x20;   ↓

执行阶段

&#x20;   ↓

成功、失败或取消

必须满足

1.runJob() 不再直接调用流水线，只负责创建 Job。

2.Worker 使用带条件的原子更新领取任务。

3\.同一个 idempotencyKey 只能存在一个有效任务。

4.Lease 过期后允许其他 Worker 重新领取。

5\.取消请求写入数据库，不只调用内存中的 AbortController。

6.Worker 在每个阶段和每批数据前检查取消状态。

7\.启动时不得把所有 running 任务直接判定失败。

8\.只有 Lease 已过期的任务才允许恢复或重新领取。

9\.自动抓取任务使用时间窗口幂等键，例如：

crawl:2026-07-21T14:00

10\.每日推送使用日期和目标作为幂等键，例如：

daily-push:2026-07-21

第一阶段可以继续保持单 Worker，但正确性必须由数据库保证，而不是由“只启动一个进程”的人为约定保证。



P0-2：聚类证据存在跨成员拼接问题

当前 candidateEvidence() 会遍历候选 Event 的多篇成员文章，然后分别取：

任意成员的标题最大相似度；

任意成员的正文最大 overlap；

任意成员的正文最大 Jaccard；

任意成员的最高身份相似度；

任意成员的最短时间距离；

任意成员的最多共享锚点。

这些指标最终被拼成同一个 Event 级证据。

随后，强匹配规则直接使用这组拼接后的证据决定是否自动合并。

错误示例

一个 Event 中有三篇文章：

成员 A：标题和新文章相似，但正文和事件不同

成员 B：正文和新文章相似，但标题和主体不同

成员 C：事件主体相似，但动作和事项不同

当前算法可能组合成：

标题相似度：取 A

正文相似度：取 B

身份相似度：取 C

时间距离：取另外一篇最近的文章

然后把它当作一条完整证据自动合并。

这是一类非常危险的误合并来源。

相反，某个旧成员存在阶段冲突时，当前的 phaseConflict ||= ... 也可能否决其他真正匹配的成员。

必须改成成员对成员的原子判断

建议增加：

type PairEvidence = {

&#x20; candidateEventId: string;

&#x20; matchedMemberArticleId: string;



&#x20; fingerprintMatch: boolean;

&#x20; eventKeyMatch: boolean;



&#x20; subjectSimilarity: number;

&#x20; actionSimilarity: number;

&#x20; objectSimilarity: number;

&#x20; identityScore: number;



&#x20; titleOverlap: number;



&#x20; charContentOverlap: number;

&#x20; charContentJaccard: number;

&#x20; tokenContentOverlap: number;

&#x20; tokenContentJaccard: number;



&#x20; daysApart: number;

&#x20; phaseConflict: boolean;

&#x20; qualifierConflict: boolean;

&#x20; sharedAnchors: string\[];

};

实现流程：

新文章

&#x20; ↓

分别与候选 Event 的每个成员计算 PairEvidence

&#x20; ↓

每个 PairEvidence 独立执行 exact / strong / ambiguous / reject 判断

&#x20; ↓

选择一个真实存在的最佳成员证据

&#x20; ↓

再把该成员的判断提升为 Event 判断

禁止在不同成员之间取各维度最大值。

审计记录必须保存：

candidateEventId

matchedMemberArticleId

全部 PairEvidence

最终选择理由

规则版本

必须增加的回归测试

1\.标题证据来自 A、正文证据来自 B、身份证据来自 C，不得自动合并。

2.A 存在阶段冲突，但 B 与新文章完全一致，应允许匹配 B。

3\.代表文章不匹配，但历史成员匹配，应正确归入 Event。

4.Event 中大量宽泛报道不得稀释或放大单个真实成员的判断。



P0-3：正文相似度的两个指标也可能来自不同表示空间

当前正文相似度同时计算：

字符 8-gram；

中文分词 token set。

最后分别返回：

overlap = max(characterOverlap, tokenOverlap)

jaccard = max(characterJaccard, tokenJaccard)

因此，最终的 overlap 和 Jaccard 可能来自两套不同的表示空间，却被后续规则视为一组同时成立的证据。

必须改为分维度输出

返回：

{

&#x20; charOverlap,

&#x20; charJaccard,

&#x20; tokenOverlap,

&#x20; tokenJaccard

}

强匹配规则必须明确要求同一表示空间中的指标同时成立，例如：

charOverlap >= X 且 charJaccard >= Y

或者：

tokenOverlap >= X 且 tokenJaccard >= Y

禁止把字符 overlap 和 token Jaccard 组合成一个不存在的证据。



P0-4：推送防重只存在于内存，无法防止重复投递

当前通过：

const inFlightPushes = new Map<string, Promise<PushArticleResult>>();

防止同一个 Event 同时推送。

它只能防止单进程内的并发，无法处理：

多实例；

进程重启；

请求超时；

发送成功后写日志前崩溃。

调用飞书 Webhook

&#x20;   ↓

收到成功响应

&#x20;   ↓

写入 PushLog

如果在飞书成功响应后、PushLog 写入前崩溃，下次任务会再次发送。

外部副作用规则

1\.发送前先创建 sending Delivery。

2\.发送成功后更新为 succeeded。

3\.发送明确失败时更新为 failed。

4\.发送请求已发出但结果无法确定时，更新为 unknown。

5.unknown 不得自动重新发送。

6\.管理员明确确认后才能重试 unknown。

7\.推送状态必须从 Delivery 派生，不能只看 Event.pushedAt。



P0-5：Event 合并时错误复制 pushedAt

当前合并 Event 时，如果源 Event 已推送而目标 Event 未推送，会把源 Event 的 pushedAt 复制到目标 Event。

但：

PushLog 仍属于源 Event；

目标 Event 的代表文章可能不同；

合并后的内容可能从未发送；

目标 Event 会被错误显示为已推送。

必须修复

1\.合并时禁止直接复制 pushedAt。

2.Delivery 历史保持关联原始 Event，不篡改历史事实。

3\.合并后重新计算目标 Event 的投递状态。

4\.如果合并改变了代表文章或卡片内容，目标 Event 应进入：

deliveryStatus = pending\_reconciliation

5\.由管理员决定是否完整重推。

6.Event 的 pushedAt 最终应从当前启用目标的 Delivery 状态派生，而不是手工维护。



三、聚类体系改建

P1-1：品牌不应取代完整事件主体

当前逻辑中，只要存在品牌，事件键主体就优先直接使用品牌，其他模型提取的参与主体会被舍弃。

这会导致以下事件过度收敛：

品牌 A 与公司 B 合作

品牌 A 与公司 C 合作

品牌 A 被监管部门处罚

品牌 A 任命某位 CEO

品牌 A 被另一家公司收购

这些事件都可能只剩：

品牌 A / 动作 / 简短事项

正确原则

品牌相同意味着：

值得召回和比较。

不意味着：

可以直接认为是同一事件。

建议事件身份结构

type EventIdentityV2 = {

&#x20; primaryEntities: Array<{

&#x20;   normalizedName: string;

&#x20;   role: 'brand' | 'company' | 'person' | 'regulator' | 'partner' | 'location';

&#x20; }>;



&#x20; action: {

&#x20;   canonical: string;

&#x20;   phase: 'planned' | 'ongoing' | 'completed' | 'unknown';

&#x20; };



&#x20; object: string;



&#x20; qualifiers: {

&#x20;   year?: number;

&#x20;   quarter?: number;

&#x20;   date?: string;

&#x20;   city?: string;

&#x20;   quantity?: string;

&#x20;   amount?: string;

&#x20;   edition?: string;

&#x20; };

};

品牌的正确用途

品牌应参与：

候选召回；

主体锚定；

排序加分；

同品牌候选展示。

品牌不得单独构成自动合并依据。



P1-2：候选召回依赖大范围 Event 扫描

当前聚类会查询时间窗口内所有 active Event，并加载每个 Event 的代表文章和最多 12 篇成员正文，没有数据库级候选数量上限。

随后才在 Node.js 中：

1\.计算快速正文提示；

2\.排序取 48 个；

3\.再计算完整正文相似度；

4\.取 15 个候选；

5\.最多给 AI 5 个。

随着 Event 数量增长，会出现：

单篇文章加载大量正文；

大量 n-gram 构建；

Node.js 事件循环阻塞；

内存峰值增长；

聚类耗时越来越长。

建议采用两阶段候选召回

第一阶段：数据库 Blocking

按以下顺序召回：

1.eventKeyHash 完全一致；

2\.内容指纹完全一致；

3\.同品牌 + 同动作 + 时间窗口；

4\.共享主体 + 同动作；

5\.同品牌 + 同地点、季度、数量或合作方；

6\.标题规范化哈希或 MinHash 接近。

只查询轻量字段，不读取完整正文。

每个召回通道设置独立上限，例如：

eventKey exact：20

contentHash exact：20

brand + action：50

entity + action：50

title signature：50

合并去重后最多保留约 100 个轻量候选。

第二阶段：精排

只对第一阶段前 20～30 个候选读取：

代表文章；

最相似历史成员；

精简正文签名；

必要时完整正文。

最终只对真正灰区候选调用 AI。



P1-3：固定 7 天窗口会造成结构性漏召回

目前事件聚类窗口固定为 7 天，推送前重复检查固定为 14 天。

不同类型事件的合理时间跨度完全不同：

即时门店新闻：3～14 天

高管任命：30～90 天

季度财报：一个季度甚至一年

融资上市：30～180 天

年度奖项：需要届次和年份，而不是短时间窗口

项目建设：预告到正式落地可能跨越数月

改建建议

1\.时间是排序和冲突特征，不应成为唯一的召回硬边界。

2\.事件键、内容指纹或明确限定词一致时，应允许跨越时间窗口召回。

3\.按 action 类型配置不同窗口。

4\.明确提取：

o年份；

o季度；

o届次；

o城市；

o门店数量；

o金额；

o合作方。

5\.不同年份、季度和届次作为强冲突。

6.“计划”和“正式完成”不应简单永久拆开，应支持事件阶段关系：

same\_event\_family

different\_event\_phase

公开和推送层可以决定是否将不同阶段作为独立新闻。



四、Event 数据一致性改建

P1-4：大量关键状态仍然是自由字符串

当前以下关键字段主要使用 String：

clusterStatus

aiStatus

reviewStatus

publicOverride

publicStatus

Event.status

clusterReviewStatus

PushLog.status

Audit action 和 decisionSource

这意味着数据库允许写入任意拼写，很多不合法状态组合只能依赖每个调用方自觉维护。

建议

1\.所有有限状态改为 Prisma enum。

2\.定义统一状态机，不允许任意更新。

3\.所有状态变化通过领域命令执行，例如：

attachArticleToEvent

moveArticleBetweenEvents

splitEvent

mergeEvents

confirmClusterReview

selectRepresentative

requestPublicationRefresh

requestDelivery

4\.禁止 Route Handler 或普通 Service 直接拼接状态字段。

5.Event 增加 version，写入时使用乐观锁。



P1-5：代表文章与 Event 成员关系缺少数据库级约束

数据库只能保证 representativeArticleId 指向某篇 Article，但不能保证该 Article 的 eventId 等于当前 Event。

因此理论上可能出现：

Event A.representativeArticleId = Article X

Article X.eventId = Event B

推荐两种方案

推荐方案：引入 EventMember

EventMember

&#x20; id

&#x20; eventId

&#x20; articleId

&#x20; membershipStatus

&#x20; isRepresentative

&#x20; joinedAt

&#x20; decisionSource

&#x20; confidence

唯一约束：

unique(eventId, articleId)

unique(articleId)

代表文章使用 EventMember 关系表达。

最小改造方案

继续保留 Article.eventId，但必须：

1\.所有代表文章变更放在事务中；

2\.事务内再次检查 Article 仍属于该 Event；

3\.使用 Event.version 防止并发覆盖；

4\.增加一致性扫描脚本；

5\.发现不一致时自动清空代表并重新计算。



P1-6：聚类事务结束后才重新计算 Event

文章创建或加入 Event 后，事务只更新 Article、Event 计数和审计，随后在事务外调用 recalculateEventById()。

如果事务提交后、重新计算前进程崩溃，可能出现：

Event 没有代表文章；

articleCount 不正确；

公开状态未刷新；

旧代表文章仍保持公开投影；

推送状态与当前成员不一致。

建议引入 EventDirty / ReconciliationTask

任何会影响 Event 的操作，在同一事务中写入：

EventDirty

&#x20; eventId

&#x20; reason

&#x20; version

&#x20; createdAt

后台 Reconciler 负责幂等重建：

articleCount

firstSeenAt

lastSeenAt

clusterReviewStatus

representativeArticleId

publicStatus

publicDateKey

publicSortAt

deliveryProjection

任务成功后删除或标记 EventDirty。

即使进程中途崩溃，下一次仍会继续修复。



P1-7：人工移动和拆分会直接把文章设为 clustered

moveArticleToEvent() 会把被移动文章直接改成 clustered。

splitEventArticles() 也会把所有被拆分文章直接改成 clustered，没有检查它们原本是否：

needs\_review

AI 未完成

聚类失败

内容无效

修复规则

1\.人工操作必须显式表达是否同时完成审核。

2\.不得通过移动或拆分隐式绕过技术门禁。

3.API 请求增加：

confirmMembership: boolean

confirmReview: boolean

4\.如果文章未完成 AI 或正文处理，禁止设为 clustered。

5\.审计记录必须保存操作前后的完整状态。

6\.所有成员移动后写入 EventDirty，由 Reconciler 重算两个 Event。



五、推送与敏感信息安全

P1-8：PushLog 持久化完整 Webhook URL

PushLog 当前保存完整 webhookUrl。

Webhook URL 本质上是可直接发送消息的凭证。一旦进入数据库，就会同时进入：

SQLite 文件；

WAL；

数据库备份；

日志分析；

开发数据库复制；

管理接口查询过程。

必须处理

1.PushLog 不再保存完整 URL。

2\.只保存：

otargetId

otargetNameSnapshot

ourlHash

3\.对现有数据库执行迁移，清空历史明文 URL。

4\.迁移完成后轮换所有飞书 Webhook。

5\.错误日志禁止输出完整 URL。



P1-9：敏感设置可以明文导出

Webhook 和 AI API Key 被标记为敏感，但仍被设置为 exportable: true。

设置导出接口明确会返回包含明文密钥的完整备份。

建议

1\.敏感信息默认不可导出。

2\.最佳方案：密钥放在环境变量或 Secret Manager。

3\.需要数据库管理时：

o使用服务端主密钥加密；

o主密钥不放数据库；

oUI 只显示“已配置”；

o修改时只允许覆盖，不能读取原值。

4\.删除普通的 reveal 明文能力。

5\.必须导出时：

o重新验证管理员身份；

o输出加密备份；

o记录安全审计；

o设置短期一次性下载；

o禁止浏览器缓存。



P1-10：管理员会话和 API Token 应分离

当前管理员会话签名直接使用 API\_TOKEN，代理验证会话后又把原始 API Token 注入请求头。

这使：

用户登录凭证；

Session 签名密钥；

内部 API Token；

成为同一个安全边界。

建议

分别使用：

ADMIN\_PASSWORD\_HASH

SESSION\_SECRET

INTERNAL\_API\_TOKEN

ENCRYPTION\_MASTER\_KEY

管理后台基于已验证 Session 授权，不需要把原始 API Token重新注入请求头。

所有写接口增加：

Origin 检查；

CSRF 防护；

操作审计；

Session 轮换；

更短的敏感操作授权窗口。



六、采集与输入安全

P1-11：存在 SSRF 风险

数据源 URL 当前只校验协议为 HTTP 或 HTTPS。

实际请求允许：

任意管理员输入的地址；

自定义请求头；

自动重定向；

访问内网或本机服务；

读取任意体积响应。

HTML Parser 还允许从 parserConfig 传入自定义 headers。

必须增加统一 safeFetch

所有解析器和正文抓取必须统一经过 safeFetch()：

1\.禁止：

olocalhost；

o127.0.0.0/8；

oRFC1918 私网；

olink-local；

o云元数据地址；

oIPv6 本地和私网；

o非 HTTP/HTTPS。

2.DNS 解析后校验所有返回 IP。

3\.每次重定向重新校验目标地址。

4\.限制重定向次数。

5\.跨域重定向时删除敏感 Header。

6.parserConfig Header 使用白名单。

7\.禁止自定义：

oHost

oCookie

oAuthorization

oProxy-Authorization

8\.限制可接受 Content-Type。

9\.对公开互联网抓取设置独立网络策略。



P1-12：URL 相对路径解析错误

当前 resolveUrl() 只处理：

已经以 http 开头的绝对地址；

以 / 开头的根路径。

对于以下常见地址会原样返回：

news/123

../article/1

./detail/2

//example.com/news/3

修复

使用：

export function resolveUrl(href: string, baseUrl: string): string {

&#x20; const resolved = new URL(href, baseUrl);



&#x20; if (!\['http:', 'https:'].includes(resolved.protocol)) {

&#x20;   throw new Error('Unsupported URL protocol');

&#x20; }



&#x20; return resolved.toString();

}

增加针对：

相对路径；

父目录；

协议相对地址；

中文编码；

fragment；

query；

的完整测试。



P1-13：URL 标准化不足，影响去重

当前 URL 标准化只做：

移除 fragment；

query 排序；

去掉末尾斜杠。

同一篇文章可能因为以下参数重复入库：

utm\_source

utm\_medium

from

source

spm

track

share

timestamp

建议

Article 同时保存：

originalUrl

canonicalUrl

canonicalUrlHash

标准化规则：

1\.提取详情页 <link rel="canonical">。

2\.移除明确的跟踪参数。

3\.标准化 hostname 大小写和默认端口。

4\.统一路径编码和末尾斜杠。

5\.支持来源级自定义规则。

6\.唯一约束放在 canonicalUrlHash。

7\.原始 URL 只用于追踪来源。



P1-14：响应体和正文存储没有硬上限

HTTP 抓取使用 response.arrayBuffer() 一次性读取完整响应，没有响应体积限制。

Article 同时保存：

完整原始 HTML；

cleanContent；

articleBody。

详情抓取时三份内容都会写入 SQLite。

建议

1\.流式读取响应。

2\.设置 HTML 最大体积，例如 3～5 MB。

3\.超过上限立即中止。

4\.只保留一份规范正文。

5\.原始 HTML：

o压缩保存；

o设置 7～30 天保留期；

o或存储在对象文件中，不放热数据库。

6\.为正文设置最大长度。

7\.聚类只读取需要的摘要、签名或限定长度正文。

8\.定期执行 VACUUM、WAL checkpoint 和容量监控。

暂时不必立即迁移 PostgreSQL。应先修复持久化语义、长事务和数据膨胀；当真实指标表明 SQLite 写锁和数据容量成为瓶颈后，再迁移数据库。



七、性能与批处理改建

P1-15：批处理先把全部待办加载进内存

AI 阶段先查询全部 pending ID，再在内存中分页。

正文处理阶段也会一次查询全部待处理文章，再进行数组切片。

改成数据库 Keyset Pagination

示例：

WHERE status = pending

&#x20; AND id > lastId

ORDER BY id

LIMIT 200

或者通过 JobItem/Lease 原子领取固定批次。

不得先读取所有 ID。



P1-16：修改评分策略会产生超长 SQLite 事务

当前评分规则变化时，会：

1\.读取全部符合条件的 Article；

2\.逐条计算；

3\.逐条 update；

4\.重建全部公开状态；

5\.将整个过程放进最长 120 秒的事务。

公开快照重建也会逐篇处理全部文章。

这会长时间占用 SQLite 写锁。

改建方案

1\.设置保存和历史重算分离。

2\.保存新规则时生成 policyVersion。

3\.创建持久化 PolicyRecomputeJob。

4\.每批处理 100～500 篇。

5\.每批独立事务。

6\.支持暂停、恢复和失败重试。

7.Article 保存：

oappliedPolicyVersion

opendingPolicyVersion

8\.全部完成后切换公开投影版本。

9\.避免逐个 Event 重算，先收集受影响 Event ID，再批量处理。

十一、最终验收标准

任务可靠性

两个 Worker 同时运行时，同一个 Job 只能被一个 Worker 领取。

Worker 在任意阶段崩溃后，Lease 到期可以继续恢复。

重启不会把其他实例正在执行的任务误判失败。

取消任务后不会继续执行下一阶段。

聚类正确性

所有自动合并证据来自同一个真实成员。

同品牌不同事件不会因为品牌相同自动合并。

标题、正文和身份证据不能跨成员拼接。

年份、季度、届次和阶段冲突可以被准确识别。

每次判断都有 matchedMemberArticleId 和完整审计证据。

黄金标注集指标可在规则变更前后自动比较。

Event 一致性

任意时刻必须满足：

event.articleCount = 实际成员数

representativeArticle 属于当前 Event

pending Event 没有公开代表

非代表 Article 不得保持 published

merged Event 不得继续作为 active 候选

空 Event 不得保持 active

公开 Event 必须存在合法代表文章

一致性脚本运行后违规数量必须为 0。

推送可靠性

同一 Event、目标和内容版本不会被并发重复投递。

发送结果未知时不自动重发。

Event 合并不会继承错误的已推送状态。

数据库、API DTO、日志和备份中不再出现完整 Webhook URL。

所有投递状态都能从 PushDelivery 重建。

安全

无法通过数据源访问 localhost、内网地址或云元数据接口。

重定向目标会重新进行安全校验。

响应体超过限制会被中止。

API Key 和 Webhook 不再支持普通明文读取和明文导出。

Session Secret 与 API Token 完全分离。



十二、禁止事项

本轮改建禁止：

1\.只调提示词和相似度阈值，不修复 PairEvidence。

2\.只增加更多 AI 调用来掩盖候选逻辑问题。

3\.直接增加向量数据库作为第一步。

4\.把单进程假设继续当作并发控制方案。

5\.使用更多内存锁解决持久化问题。

6\.在 SQLite 中执行长达数分钟的全库事务。

7\.合并 Event 时直接复制公开或推送状态。

8\.在 PushLog、错误日志或 DTO 中记录完整 Webhook。

9\.没有 migration 就修改 Prisma schema。

10\.没有真实标注集就宣称聚类准确率得到提升。

改建的核心目标不是让代码看起来更复杂，而是建立四个可靠基础：

任务可恢复

聚类可解释

状态可重建

效果可测量
