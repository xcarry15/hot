# Hot2 第一性原理架构精简与实用化改造方案

> 文档定位：后续代码改造的唯一实施控制文档  
> 复核日期：2026-07-19  
> 复核基线：当前工作区代码、`README.md`、`AGENTS.md`、Prisma Schema 及现有性能审查文档  
> 核心约束：不兼容历史业务数据，Schema 调整后重新下载、重新采集

## 1. 最终结论

Hot2 当前的技术路线基本正确，不需要更换框架或拆成微服务。最适合项目现阶段的架构仍然是：

```text
Next.js 模块化单体
  + Prisma
  + SQLite WAL
  + PM2 单实例
  + 单 Job 执行器
```

当前真正影响实用性、稳定性和维护成本的，不是技术栈，而是以下四类问题：

1. **业务决策仍部分落在 Article，最终公开与推送却以 Event 为单位，形成重复真相。**
2. **后台仍以“文章未人工点过”为待办，制造了大量并不需要管理员处理的工作。**
3. **设置页中的 Dashboard、人工建议、任务历史、推送日志与另外两个工作台职责重叠。**
4. **派生状态同步、缓存失效、长耗时重建分散在多个 Service，修改一个业务规则需要触碰很多文件。**

最终改造不追求代码量最少，而追求：

- 管理员只处理真正的异常和高价值决策；
- 同一业务问题只有一个权威答案；
- 常规操作结果可预测，不依赖隐含联动；
- 故障可以定位、重试和审计；
- 新功能能沿现有边界增加，不必复制一套状态判断；
- 删除低价值功能后，核心代码更少、更容易理解。

## 2. 从第一性原理重新定义项目

### 2.1 用户真正需要的结果

项目的最终产物不是 Article，也不是抓取日志，而是：

> 将不同数据源的报道自动整理成可信、可审核、可发布、可推送的行业 Event，并让管理员只处理机器无法可靠决定的部分。

因此主链路应固定为：

```text
Source
  → Article 采集与内容加工
  → Event 聚类
  → Event 级人工决策
  → Event 级公开
  → Event × 推送目标投递
```

### 2.2 必须保留的五类事实

| 事实类型 | 权威模型 | 说明 |
| --- | --- | --- |
| 数据来源事实 | Source | 来源配置、启停、健康状态 |
| 原始内容与 AI 事实 | Article | 原文、正文、AI 原始结果、人工字段覆盖 |
| 事件与编辑决策事实 | Event | 成员、代表文章、重要性、公开决策、公开统计 |
| 任务执行事实 | Job | 批任务、单篇工作流、维护重建、进度、失败、取消 |
| 不可变审计事实 | FetchLog / PushLog / EventClusterAudit | 发生过什么，不作为当前状态的唯一判断依据 |

### 2.3 输入、当前状态、审计必须分开

- **输入**：管理员设置的公开覆盖、重要性、代表文章、推送目标配置。
- **当前状态**：Event 当前是否公开、各推送目标是否成功、Job 当前阶段。
- **审计**：曾经发布、推送、聚类或人工修改过什么。

允许为读取性能持久化“派生投影”，但必须明确：

1. 哪个字段是权威事实；
2. 哪个 Service 唯一负责更新投影；
3. 其它模块不得自行推断并写入同一状态。

## 3. 当前代码中已经完成、不要重复实施的内容

以下内容已存在于当前工作区，后续 AI 应先确认代码仍然存在，再继续后面的改造：

- SQLite 启动时初始化 WAL、`synchronous=NORMAL`、`busy_timeout=5000`、外键与 `PRAGMA optimize`；
- 新增 `db:optimize` 及部署文档同步；
- 公开列表已增加 `Event.publicDateKey/publicSortAt` 和有限日期查询；
- 公开列表缓存已有容量上限；
- 公开详情 metadata 与页面读取已进行请求级复用；
- 抓取记录已拆出 Job 轻快照，运行中只高频轮询轻数据；
- 页面隐藏时暂停抓取记录和 Dashboard 周期请求；
- 后台顶层页面和设置子模块已使用动态加载；
- 收件箱文章详情已有短时请求合并和预取；
- 技术工作队列已有 5 秒短缓存；
- 浏览量已按批次落库；
- 关键词候选单篇处理已从逐个查询改为批量预取后事务写入；
- `needs_review` 可继续 AI 分析，但公开和推送已要求 `clusterStatus=clustered`。

这些改动改善了当前性能，但没有解决领域真相重复和后台职责问题。本方案从这些结果继续推进。

## 4. 目标领域模型

### 4.1 Article：只表达原始报道和内容校准

Article 应保留：

- Source 关联、URL、标题、发布时间；
- `rawContent / articleBody / cleanContent / contentHash`；
- fetch、cluster、AI 阶段状态及重试字段；
- `eventId / eventKey`；
- AI 原始结果、最终有效内容字段、评分字段；
- `aiSnapshot / manualOverrides / manualCorrectedAt`；
- Article 对应的聚类审计。

Article 应删除以下最终业务决策字段：

- `reviewStatus`
- `reviewReasonTags`
- `reviewedAt`
- `publicOverride`
- `pinUntil`
- `publicStatus`
- `publicPublishedAt`
- `publicRevokedAt`
- `publicPublicationReason`
- `publicPublicationEvaluatedAt`
- `publicContentUpdatedAt`
- `viewCount`
- `originalClickCount`

原因：这些字段描述的是 Event 的编辑决策、公开结果和公开页面行为，不属于某一个来源 Article。

### 4.2 Event：成为编辑、公开和统计的业务聚合根

Event 应保留或新增：

```text
成员与聚类
  status
  mergedIntoId
  representativeArticleId
  representativeManual
  clusterKey
  firstSeenAt / lastSeenAt / articleCount

编辑决策
  editorialClass        auto | important | general | irrelevant
  reviewReasonTags
  reviewedAt
  publicOverride        auto | public | hidden
  pinUntil

公开投影
  publicStatus          unpublished | published | revoked
  publicReason
  publicPublishedAt
  publicRevokedAt
  publicContentUpdatedAt
  publicDateKey
  publicSortAt

公开统计
  viewCount
  originalClickCount
```

规则固定为：

- `auto`：按评分、软文和来源规则自动决定；
- `important`：同时设置 `publicOverride=public` 并置顶；
- `general`：恢复 `publicOverride=auto`；
- `irrelevant`：同时设置 `publicOverride=hidden`；
- 人工也可以单独修改 `publicOverride`，但 UI 必须明确显示实际结果；
- 代表 Article 变化时，Event 的公开状态、公开内容时间和排序投影统一重算；
- 浏览量和原文点击始终累计到 Event，不因代表 Article 更换而丢失。

删除以下可配置映射，避免“点了重要但结果由另一个设置决定”：

- important → public 的可配置规则；
- general → auto 的可配置规则；
- irrelevant → hidden 的可配置规则。

这三条应成为固定产品语义。

### 4.3 EventPushTargetState：当前投递状态唯一真相

新增当前状态表：

```prisma
model EventPushTargetState {
  eventId       String
  targetId      String
  status        PushTargetStatus
  retryCount    Int
  nextRetryAt   DateTime?
  lastAttemptAt DateTime?
  succeededAt   DateTime?
  error         String
  updatedAt     DateTime

  event Event @relation(...)

  @@id([eventId, targetId])
  @@index([status, nextRetryAt])
}
```

同时调整 Webhook 配置：

```ts
interface WebhookConfig {
  id: string;       // 稳定目标标识，不能使用 remark 或 URL 充当长期身份
  url: string;
  remark: string;
  enabled: boolean;
}
```

PushLog 继续保留，但只作为不可变历史：

- 增加 `targetId`；
- 保留发送时 URL、备注、代表 Article 快照；
- 每次投递后同时 upsert `EventPushTargetState`；
- 当前是否失败、是否需要重试、当前启用目标是否全部成功，只读取状态表；
- 被删除或停用的目标不参与当前完成判断，历史 PushLog 仍保留；
- 删除 `Event.nextPushRetryAt`；
- `Event.pushedAt` 可删除。若保留为只读投影，应改名为 `pushCompletedAt`，并且只能由 delivery service 根据当前启用目标统一更新，不能作为权威判断。

### 4.4 Job：只表达真实存在的任务状态

删除历史兼容状态：

- `JobType.fastProcess`
- `JobStatus.pending`

目标状态：

```text
JobType:
  full | collect | process | cluster | ai | push | maintenance

JobStatus:
  running | completed | failed | cancelled
```

建议增加可查询列，避免 Job 页面读取 JSON 后在 Node.js 中过滤：

- `trigger`: auto / manual / system
- `scope`: all / source / article / event
- `targetId`
- `operation`: maintenance 类型的具体操作

`payload/result` 继续保留详细快照，但不再承担常用筛选字段职责。

管理员停止任务应落为 `cancelled`，不能再伪装成普通失败。

### 4.5 删除低价值模型

删除：

- `TuningSuggestion`
- `InboxSnapshot`

原因：

- 两三条人工样本就建议调整全局阈值，不具备可靠统计意义；
- GET Dashboard 时生成并写入建议，违反只读接口预期；
- “所有 unreviewed Article 的积压趋势”本身不是有效管理指标；
- 删除后可同时移除一组 API、页面状态、自动写入和全表统计。

## 5. 后台产品结构重组

后台继续只保留三个顶层入口，但职责重新冻结。

### 5.1 情报收件箱：Event 级业务工作台

默认列表从 Article 改为 Event，一行代表一个 Event。

建议视图：

1. **待处理**：只显示真正需要人工判断的 Event；
2. **全部事件**：浏览所有 Event；
3. **人工修改**：显示做过人工内容覆盖或编辑决策的 Event；
4. **已隐藏**：便于恢复误隐藏内容。

“待处理”只能包含：

- 代表 Article 或成员存在 `clusterStatus=needs_review`；
- 代表 Article AI 已完成但置信度低于阈值；
- Event 结构存在无法自动修复的不一致；
- 管理员主动标记的待复核事项（如果后续确有需求再增加）。

以下内容不能算人工待办：

- 普通 `unreviewed`；
- fetch、cluster、AI、push 技术失败；
- 仅仅因为新采集但没有异常。

Event 详情应分成三个稳定区域：

- **事件决策**：重要/普通/无关、公开覆盖、置顶、人工推送；
- **内容校准**：默认编辑代表 Article 的摘要、品牌、分类、标签、关键点和评分；
- **来源与聚类**：成员列表、移动、拆分、合并、确认独立、指定代表 Article、审计依据。

删除或降级冗余视图：

- 删除单独的“代表文章”列表视图，因为 Event 行天然展示代表 Article；
- “多来源”保留为筛选条件，不作为独立工作流；
- 技术失败不在收件箱重复展示，只提供跳转到抓取记录的链接；
- Article 全量列表仅作为 Event 详情内的成员浏览，不再作为默认后台首页。

建议 API：

```text
GET    /api/events?view=attention|all|edited|hidden&cursor=...
GET    /api/events/[id]
PATCH  /api/events/[id]/editorial
PATCH  /api/articles/[id]               # 只修改 Article 内容校准字段
POST   /api/events/[id]/move|split|...
POST   /api/events/merge
POST   /api/events/[id]/push
```

### 5.2 抓取记录：技术运行与恢复工作台

保留并整合以下高价值能力：

- 当前 Job、阶段、进度、耗时、取消；
- 技术失败队列：process / cluster / AI / push；
- Source 健康、最近失败、断路器、重试；
- 最近 Job 历史，服务端筛选和分页；
- PushLog 与当前目标投递状态；
- 未入库记录及强制恢复；
- 深链到情报收件箱的对应 Event 或 Article。

抓取记录不负责：

- 内容评分编辑；
- 重要/普通/无关决策；
- Event 合并、拆分和代表 Article 选择；
- 展示评分构成、软文概率、内容分类等业务校准字段。

当前设置页 Dashboard 中仍有价值的能力迁移到抓取记录：

- Source 运行质量表；
- Job 历史；
- PushLog；
- 当前日的简要采集/处理结果。

历史趋势只在管理员主动选择 7 天或 30 天时按需读取，不自动轮询。

### 5.3 设置：只负责配置

删除设置页“概览”默认页，设置页不再承担第四个业务工作台。

建议分组：

```text
数据源
关键词
AI 模型
提示词
发布与推送
系统
```

其中：

- “公开”和“推送”合并为输出策略；
- “账户”和“数据工具”归入系统；
- Source CRUD、测试和启停留在设置；Source 健康和恢复留在抓取记录；
- 使用 `/admin/settings/[section]` 或等价的真实路由，保留顶层导航只有“设置”。

### 5.4 精简数据清理功能

当前设置页存在大量会制造部分状态的清理按钮。考虑项目允许重新采集，最终只建议保留：

- 配置导入/导出；
- 数据库优化；
- 清空全部业务数据并重新采集；
- 必要时导出诊断信息。

删除或迁移：

- 删除低质量文章；
- 删除已推送文章；
- 单独清空去重日志；
- 单独清空抓取日志；
- 设置页直接重置全部 AI；
- 设置页直接重置失败 AI。

失败恢复应在抓取记录完成；全量重新分析应作为 Job 显示进度，不能作为设置页同步请求。

## 6. 服务端代码边界收敛

### 6.1 建立一个 Event 一致性入口

当前 `refreshPublicPublication()` 和 `invalidatePublicArticleCache()` 被大量 Service 手工组合调用。应收敛为一个应用级入口，例如：

```ts
reconcileEventForArticle(articleId, {
  contentChanged?: boolean;
  membershipChanged?: boolean;
  editorialChanged?: boolean;
})

reconcileEvent(eventId, reason)
```

该入口统一负责：

1. 校验成员与代表 Article；
2. 必要时重选代表 Article；
3. 计算 Event 时间范围、成员数和 clusterKey；
4. 根据 Event 编辑输入和代表 Article 计算公开资格；
5. 更新 Event 公开投影；
6. 最后统一失效公开缓存。

其它模块不得再自行执行“更新 Article → 刷新公开 → 重算 Event → 再失效缓存”的不同组合。

### 6.2 策略函数必须纯化

以下规则放到纯函数中，输入完整、输出确定，不读数据库：

- 代表 Article 排序；
- Event 公开资格和原因；
- Event 编辑动作到字段更新的映射；
- 推送资格；
- Article 流水线状态投影；
- 人工待办原因投影。

Service 负责读取、事务和写入，组件与 Route Handler 不复制规则。

### 6.3 拆分 execution.ts，但不增加新框架

保留 `runJob()` 为唯一任务入口，将约 500 行执行器按用途拆开：

```text
src/lib/jobs/runtime.ts                 # Job 生命周期、互斥、心跳、取消
src/lib/jobs/pipeline-executors.ts      # full/collect/process/cluster/ai/push
src/lib/jobs/article-workflow.ts        # 单篇 retry/regenerate
src/lib/jobs/maintenance-executor.ts    # 重算评分、重建公开、候选词生成
```

不要引入消息队列、独立 Worker 或通用工作流框架。

### 6.4 缩小全局 Mutation Guard

当前所有写操作共用一个全局门禁，会导致抓取运行时普通编辑、设置和其它短写入全部被拒绝。

目标规则：

- 后台批量 Job 同一时间只能有一个；
- destructive maintenance 必须等待无 Job；
- Event 合并、拆分、移动、代表 Article 变更使用事务并按相关 Event 串行；
- 普通 Article 内容校准和 Event 编辑允许短事务执行；
- 会触发全量重建的设置修改只保存配置，重建交给 maintenance Job；
- 只读 GET 绝不能为了生成建议而占用写入门禁。

不需要建立分布式锁，因为生产仍是单实例。

### 6.5 Source Service 合并边界

当前 `source-actions.ts` 与 `source-service.ts` 按历史形成边界。统一为清晰的查询与命令：

```text
src/lib/source/query.ts
src/lib/source/command.ts
src/lib/source/recovery.ts
```

规模不大时也可以合并为一个 `source-service.ts`。禁止为了形式再建立通用 Repository。

## 8. 仍需完成的性能与稳定性改造

### P0：公开读取不能残留全量扫描和静默截断

当前公开列表已经不再读取全库正文，但仍使用“每个日期最多 250 条”的硬上限；公开详情的上一篇/下一篇仍读取全部已发布 Event 后在内存排序。

改造要求：

- 列表使用 `(publicDateKey, publicSortAt, id)` 稳定游标；
- 不允许以固定 250 条静默丢失同日内容；
- 上一篇和下一篇各使用一条有界查询；
- 公开详情按 Event id 单条读取；
- sitemap 可以全量读取 id，但要只选择必要字段并评估分批；
- 公开查询只判断 `Event.publicStatus`，不再同时要求 Article 公开状态。

### P0：设置保存不能执行分钟级长事务

当前评分权重或公开策略变化可能在设置请求中逐篇重算并重建发布状态。

改造要求：

- 设置保存只校验并写入设置；
- 需要重算时创建 `maintenance` Job；
- Job 分批处理，例如每批 100～200 条；
- Job 页面显示操作类型、进度和失败；
- 重建失败不回滚已保存设置，但必须标记投影版本落后并允许重试；
- 或采用“配置版本 + 投影版本”判断是否需要重建。

### P1：Event 聚类候选必须有硬上限

当前每篇文章会读取时间窗口内全部 active Event，再在内存排序。

改造要求：

- Event 持久化 `clusterKey`；
- 先按 key 和时间窗精确查询；
- 再按代表标题或最近 Event 做有限回退；
- 单篇候选 Event 和候选成员都必须有数据库 `take`；
- AI 只判断少量模糊候选；
- 聚类审计保留选择依据。

### P1：技术队列读取当前状态，不扫描 PushLog 历史

`EventPushTargetState` 完成后：

- 技术队列只读取失败 Article、异常 Source 和失败 target state；
- 导航只读取 count，不加载完整队列；
- 抓取记录按页读取技术项；
- PushLog 只在用户打开历史详情时查询。

### P1：关键词候选移出 Article 同步主链路

关键词召回有价值，但不应让每篇未命中文章同步生成和写入最多 24 个 n-gram。

改造要求：

- process 阶段只保存 `DiscardedItem`；
- 候选词由管理员主动分析最近未命中数据，或由 maintenance Job 批量生成；
- 优先提取品牌、专名和高频短语，减少无意义 2～6 字全组合；
- KeywordCandidate 可以保留，用于记录采用和忽略状态；
- 批量批准后最多恢复固定数量的最近条目，剩余内容在下一轮采集中重新获取。

### P1：Source 统计按需聚合

从旧 Dashboard 迁移到抓取记录的 Source 质量数据应：

- 默认只读取今天的轻量汇总；
- 7 天/30 天趋势按需请求；
- SQL 聚合后返回，不把全部 Article、DiscardedItem、FetchLog 拉入 Node.js；
- Job 历史使用数据库 count + 分页，不读取最近 500 条后在内存筛选；
- 页面隐藏时不刷新历史统计。

### P2：大组件按业务区拆分

完成页面职责删除后，再拆分：

```text
intelligence-inbox.tsx
  → event-inbox-page
  → event-list
  → event-detail
  → editorial-panel
  → article-calibration-panel
  → cluster-panel

crawl-log-tab.tsx
  → job-summary
  → technical-queue
  → source-health
  → job-history
  → delivery-history
```

拆分目标是缩小状态影响范围和业务依赖，不以“每个文件低于多少行”为目标。避免创建只转发 props、没有独立职责的小组件。

## 10. 每阶段验收条件

### 10.1 业务正确性

- 一个 Event 的编辑、公开和统计状态只有 Event 一份权威数据；
- 更换代表 Article 不会丢失公开状态、浏览量或推送状态；
- `needs_review` 可以分析，但绝不公开和推送；
- 普通新 Event 不会自动成为人工待办；
- 推送目标新增、停用、修改备注或 URL 后，当前完成状态可解释；
- retry_failed 只补当前失败目标，repush_all 明确重推全部启用目标；
- 被取消的 Job 显示 cancelled，不显示为系统失败。

### 10.2 管理员体验

- 进入后台首先看到需要决策的 Event，而不是所有新 Article；
- 技术错误只在抓取记录处理；
- 设置页只展示配置，不自动运行统计或写入建议；
- 每种失败都有原因、可重试时间和唯一恢复入口；
- 常规设置保存快速返回，全量重建在 Job 中可见；
- 一个业务动作不需要管理员跨两个页面重复处理。

### 10.3 性能与稳定性

| 场景 | 目标 |
| --- | --- |
| Event 收件箱普通分页，100k Article | P95 < 200 ms |
| 公开首页，10k Event | P95 < 200 ms |
| 公开详情，10k Event | P95 < 150 ms，不扫描全部 Event |
| Job 轻快照 | P95 < 100 ms，响应 < 20 KB |
| 技术待办导航计数 | P95 < 100 ms |
| 抓取记录 Source 明细 | P95 < 300 ms，服务端分页 |
| Source 30 天趋势，100k Article | P95 < 500 ms，SQL 聚合 |
| 常规设置保存 | < 1 秒，不执行全库重建 |
| SQLite | WAL，常规运行无 `database is locked` |

### 10.4 工程质量

- Route Handler 只解析输入和返回响应；
- 组件不包含公开、推送、聚类资格规则；
- 同一规则不存在两套 where 条件；
- 结构改动同步 Prisma migration、README、AGENTS 和 bat 部署说明；
- 删除模型时同时删除 API、客户端、组件和测试中的引用；
- 不增加历史数据兼容、双写、回填脚本或旧字段 fallback。

## 12. 明确不做

当前阶段不实施：

- Redis；
- RabbitMQ、Kafka 或独立 Worker；
- 微服务拆分；
- 多 PM2 实例；
- 立即迁移 PostgreSQL；
- Embedding/向量数据库；
- 浏览器内存任务队列；
- 通用 Repository、DDD 框架或事件溯源框架；
- 为旧 Article/Event/Job 数据保留兼容分支；
- 无性能证据的大范围 React memo 化；
- 只为降低文件行数进行机械拆分。

只有当完成本方案后仍有明确、可测量的容量瓶颈，才重新评估数据库或基础设施升级。

## 13. 给后续实施 AI 的工作规则

1. 开始每个阶段前先重新读取当前代码和 Git diff，不把本文中的文件状态当成永远不变；
2. 按阶段实施，不一次混合 Schema、后台重做和性能优化；
3. 当前工作区已有大量未提交修改，必须保留用户现有改动；
4. 优先删除重复能力，再拆组件；
5. 新增字段前先判断它是输入、当前状态、投影还是审计；
6. 同一业务答案只能有一个权威来源；
7. 任何全量操作必须进入 Job，并可查看进度、失败和取消结果；
8. 不兼容历史数据，不写双读、双写或回填逻辑；
9. 每阶段同步文档和对应回归测试；
10. 完成一个阶段后更新本文复选框和“当前完成状态”，再开始下一阶段。
