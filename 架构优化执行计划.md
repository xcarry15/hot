# Hot2-S 后续优化执行计划

## 当前结论

本轮高风险代码问题已经修复并完成对抗性复核。项目规模较小，当前采用单进程 + SQLite 的架构即可满足个人和小团队使用，不再为了“架构完整”提前引入复杂分层、队列、多实例或通用基础设施。

接下来以功能好用为主：只有在出现数据错误、安全风险、状态残留，或现有结构明显阻碍功能时，才做局部重构。

## 已完成的必要修复

- [x] 统一 AI 重置、重复状态和失效引用清理。
- [x] 修复重新抓取、人工恢复、删除文章后的状态残留。
- [x] 删除文章改为事务，并解除其他文章的失效重复引用。
- [x] 浏览量和原文点击不再污染公开内容更新时间与 sitemap。
- [x] 新增 `Article.publicContentUpdatedAt`，明确公开内容时间事实源。
- [x] 后台会话改为带服务端过期校验的签名 Cookie，不兼容旧 Token Cookie。
- [x] 统一固定窗口限流，修复可伪造 IP 信任和内存容量风险。
- [x] 修复金额归一化、清空失败后的自动采集开关恢复，以及删除文章后的公开缓存失效。
- [x] 已完成类型检查、ESLint、完整 Vitest、生产构建、迁移状态和差异检查。

## 完成标准

- 功能主流程可直接完成，空数据、失败和重复状态都有清晰反馈。
- 业务状态变更不留下残留数据，不绕过现有 Service 规则。
- 高风险改动执行针对性验证；交付前至少确认类型、Lint 和差异格式无误。
- 不为历史业务数据增加兼容分支；需要时直接重新采集。
