# Hot2-S 聚类可视化与人工校准改造方案

> 文档用途：作为后续 AI 或开发人员实施改造的直接依据。实施前必须重新检查当前代码，避免覆盖已有变更或重复实现。

> 数据处理原则：本次改造不兼容、不迁移历史 Article、Event、PushLog 或 EventClusterAudit 业务数据。实施完成后清空旧业务数据并重新采集；不得为了保留旧数据增加兼容字段、回填脚本、双读逻辑或临时分支。

## 1. 背景

项目已经建立 Article → Event 架构：

```text
Article：原始文章、AI 分析、人工校准
   ↓ 事件归属
Event：同一新闻事件、代表文章、公开状态、唯一推送状态
```

Event 已经是公开和推送的唯一门禁，但管理员目前仍难以快速回答：

1. 文章是否已经完成聚类？
2. 文章为什么被归入当前 Event？
3. 当前 Event 中是否混入了不属于同一事件的文章？
4. 代表文章为什么是这一篇，是否需要更换？
5. 聚类错误后应该确认、移动、拆分还是重新聚类？

当前主要问题不是后端完全缺少能力，而是聚类事实、判断依据和人工纠错操作没有形成面向管理员的完整工作流。

## 2. 改造目标

本次改造只解决以下目标：

- 管理员能够在抓取记录页快速发现聚类流程异常。
- 管理员能够在情报收件箱判断聚类结果是否正确。
- 管理员无需理解或手工输入 Event ID，即可完成事件纠错。
- 聚类失败、业务不确定和正常聚类具有清晰、不同的处理方式。
- 人工确认、移动、拆分、合并和代表文章修改均写入审计记录。
- 保持项目简单，不建设独立 Event 管理系统或复杂数据平台。

项目规模少于 20 个数据源、每天少于 200 篇文章，设计应优先保证清晰、可靠和人工操作效率。

## 3. 页面职责

两个页面都展示聚类信息，但承担不同职责。

```text
抓取记录页：聚类流程是否正常、哪里卡住了
情报收件箱：聚类结果是否正确、如何人工修正
```

### 3.1 抓取记录页

定位为运行监控和异常入口：

- 展示 `采集 → 处理 → 聚类 → AI → 推送` 全流程状态。
- 筛选待聚类、聚类失败和待复核文章。
- 展示失败原因、重试次数和下次重试时间。
- 支持单独运行聚类阶段。
- 异常文章提供“去校准”入口，定位到情报收件箱对应 Article。

抓取记录页不负责：

- 合并 Event。
- 移动 Article 到其他 Event。
- 指定代表 Article。
- 展示完整聚类证据。
- 复杂比较同事件成员。

### 3.2 情报收件箱

定位为 Article 人工校准和 Event 人工纠错主工作台：

- 左侧全量 Article 表格展示聚类概况。
- 右侧展示当前聚类结果、判断依据和同事件成员。
- 支持确认、移动、独立、合并和指定代表文章。
- 聚类失败时支持重新聚类或人工归属。

## 4. 抓取记录页改造

### 4.1 保留的聚类信息

单篇文章步骤必须包含：

```text
采集 | 处理 | 聚类 | AI | 推送
```

状态规则：

- 处理未完成：聚类为阻塞。
- `clusterStatus=pending`：聚类为待处理。
- `clusterStatus=failed`：聚类为失败。
- `clusterStatus=clustered`：聚类完成。
- `clusterStatus=needs_review`：聚类步骤完成，同时显示“待复核”提示。
- 聚类未完成或失败：AI 和推送必须显示阻塞。

### 4.2 筛选

必须提供：

- 待聚类。
- 聚类失败。
- 待复核。
- 含失败，且必须包含聚类失败。

### 4.3 异常信息

聚类失败时展示：

- `clusterError`。
- `clusterRetryCount`。
- `nextClusterRetryAt`。
- 是否已经耗尽自动重试。

建议显示示例：

```text
聚类失败
原因：AI 返回内容不完整
已重试：5 次
自动重试：已停止
```

### 4.4 操作

保留：

- 批量运行事件聚类阶段。
- 刷新状态。

建议增加：

- 聚类失败文章的“重新聚类”。
- 待复核和聚类失败文章的“去校准”。

“去校准”应打开情报收件箱并自动选中对应 Article，不应只跳到页面顶部。

推荐使用稳定查询参数：

```text
/admin?tab=inbox&articleId=<articleId>&panel=cluster
```

如果当前后台使用其他 tab 参数规范，应复用现有规范，不得另建第二套导航状态。

## 5. 情报收件箱左侧列表

### 5.1 增加明确的事件列

左侧必须能在不打开详情的情况下看到聚类概况。

建议将当前“核验”与“事件”分开表达。事件列显示：

- 待聚类。
- 聚类失败。
- 待复核。
- 单篇事件。
- `N 来源`。
- `N 来源 · 代表`。
- `N 来源 · 人工代表`。

颜色规则：

- 聚类失败：红色。
- 待复核：琥珀色。
- 正常事件：中性灰色。
- 当前代表文章：主色或绿色，但避免大面积高亮。

不要把完整 Event ID 作为主要显示内容。可以在悬浮提示或复制按钮中提供。

### 5.2 视图筛选

在现有全量 Article 视图基础上增加：

- 待聚类。
- 聚类失败。
- 聚类待复核。
- 多来源事件。
- 代表文章。

前三项优先级最高。不得改变“全部文章”为默认视图。

### 5.3 列表一致性

完成事件操作后必须同步刷新：

- 当前 Article 行。
- 原 Event 其他成员行。
- 目标 Event 成员行。
- 事件来源数量。
- 代表文章标记。
- 聚类状态。

如果相关行不在当前分页，可以刷新当前页，但不得错误清空当前选中 Article。

## 6. 情报收件箱右侧聚类校准台

右侧文章详情中新增独立的“事件聚类”区域，位置应高于 AI 洞察和人工归类操作。

### 6.1 聚类概况

展示：

- 聚类状态。
- 当前 Event 的代表标题。
- 当前 Event 来源数量。
- 当前 Article 是否为代表文章。
- 代表文章是自动选择还是人工指定。
- `eventKey`。
- 聚类时间。
- Event 是否已经公开。
- Event 是否已经推送。

示例：

```text
聚类状态：待复核
当前事件：瑞幸发布 2026 年第二季度财报
来源数量：3
归属方式：AI 判断
置信度：72%
事件键：瑞幸咖啡 | 发布财报 | 2026 Q2
代表文章：时代财经（自动选择）
```

### 6.2 聚类判断依据

读取最近一次与当前归属相关的 `EventClusterAudit`，展示：

- `actor`：system 或 admin。
- `action`：create、attach、fallback_create、merge、split、confirm、move、representative_change 等。
- `decisionSource`：exact、rule、ai、admin。
- `confidence`。
- `candidateEventId` 对应的代表文章标题。
- `evidence` 中的有效字段。
- `ruleVersion`。
- AI 理由。
- 操作时间。

证据应翻译成人类可读内容：

```text
判断方式：AI
标题重叠：76%
事件键一致：是
内容指纹一致：否
AI 理由：主体、财报周期和发布动作一致
规则版本：event-cluster-v1
```

正常聚类默认折叠；`needs_review` 和 `failed` 默认展开。

不得把原始 JSON 直接作为唯一展示方式。可以在高级详情中提供原始 JSON。

### 6.3 同事件成员比较

将当前简单来源列表改为紧凑比较表格：

| 代表 | 文章 | 来源 | 时间 | 总分 | 相关度 | 人工归类 | 内容状态 | 操作 |
|---|---|---|---|---:|---:|---|---|---|

每行支持：

- 在当前校准台切换查看该 Article。
- 设为代表文章。
- 移到其他 Event。
- 独立为新 Event。

需要明确标记：

- 当前代表文章。
- 自动代表或人工代表。
- 软文。
- 聚类待复核。
- 已公开或已推送 Event。

## 7. 人工纠错操作

### 7.1 确认属于当前事件

适用于 `clusterStatus=needs_review`。

操作结果：

- Article 保持当前 `eventId`。
- `clusterStatus` 改为 `clustered`。
- 清理不再适用的聚类错误和重试时间。
- 写入 `actor=admin`、`action=confirm` 的 EventClusterAudit。
- 审计中保留原系统判断证据。
- 刷新公开状态和页面数据。

这是必要操作，否则管理员看过文章后无法消除“待复核”状态。

### 7.2 移到其他事件

这是最高优先级的新操作。

用户点击后打开事件搜索选择器，支持搜索：

- Event 代表文章标题。
- Event 下任一 Article 标题。
- 品牌。
- 来源。
- Event ID。

候选项展示：

```text
瑞幸发布 2026 年第二季度财报
3 个来源 · 代表：时代财经 · 7月17日 · 已推送
```

选择后一次完成：

```text
当前 Article 从原 Event 移到目标 Event
```

服务端必须在一个业务操作中：

1. 校验原 Event、目标 Event 均有效且不同。
2. 校验目标 Event 为 active。
3. 更新 Article.eventId 和聚类状态。
4. 重算原 Event。
5. 重算目标 Event。
6. 必要时删除空 Event。
7. 刷新两个 Event 的代表文章和公开状态。
8. 保持两个 Event 已有的推送事实，不允许隐式清空 `pushedAt`。
9. 写入 `action=move`、`decisionSource=admin` 审计。

管理员不需要先拆分再合并，也不需要复制 Event ID。

### 7.3 独立为新事件

替换当前容易误解的“拆分”文案。

适用于：

- 当前 Event 包含多篇 Article。
- 管理员确认当前 Article 是独立事件。

操作结果：

- 创建新 Event。
- 将选中 Article 移入新 Event。
- 新 Event 默认未推送。
- 新 Event 根据代表 Article 重新评估公开状态。
- 重算原 Event。
- 写入 `action=split` 审计。

确认提示必须说明：

> 该文章将成为一个新的独立事件，并重新参与公开和推送资格判断。

### 7.4 合并整个事件

保留事件级合并，但删除手工输入 Event ID 的主要交互。

使用与“移到其他事件”相同的搜索选择器选择目标 Event。

确认提示必须展示：

- 当前 Event 标题。
- 当前 Event Article 数量。
- 目标 Event 标题。
- 目标 Event Article 数量。
- 两个 Event 是否已公开或已推送。

确认文案：

> 当前事件中的 N 篇文章将全部移动到目标事件。历史推送不会撤回，也不会自动补发。

### 7.5 指定代表文章

只能选择当前 Event 成员。

操作后：

- `representativeArticleId` 指向选择的 Article。
- `representativeManual=true`。
- 自动重算不得覆盖人工代表。
- 刷新 Event 公开状态。
- 不改变 Event URL。
- 不修改历史 PushLog。
- 不因代表文章变化再次普通推送。
- 写入 `representative_change` 审计。

页面应解释自动代表规则：

```text
人工指定 > 人工归类重要 > 总分 > 相关度 > 正文完整度 > 发布时间
```

## 8. 聚类失败处理

`clusterStatus=failed` 时，不显示“确认当前事件”，因为 Article 通常没有有效 Event。

应提供：

- 重新聚类。
- 移到已有事件。
- 手动创建新事件。

### 8.1 重新聚类

单篇重新聚类应：

- 清理 `clusterError`。
- 将失败状态恢复为可执行状态。
- 重置或按明确规则处理 `clusterRetryCount`。
- 运行单篇聚类，而不是要求管理员重新跑整个全流程。
- 成功后按正常门禁进入 AI。

### 8.2 手动创建新事件

用于管理员确认没有可匹配 Event 的情况：

- 创建 active Event。
- 当前 Article 成为成员和默认代表文章。
- `clusterStatus=clustered`。
- 写入 `action=manual_create` 审计。
- 随后正常进入 AI、公开和推送判断。

## 9. Event 搜索选择器

不建设复杂 Event 管理页，只提供可复用的轻量搜索弹窗。

### 9.1 搜索范围

- 默认查询最近 30 天 active Event。
- 最多返回 20 条。
- 支持代表标题、成员标题、品牌和 Event ID。
- 排除当前 Event。

### 9.2 返回字段

- Event ID。
- 代表 Article ID 和标题。
- 来源数量。
- 首次和最近发现时间。
- 代表来源。
- 代表文章评分和相关度。
- 是否已公开。
- 是否已推送。

### 9.3 排序

优先级建议：

1. 标题精确匹配。
2. 品牌匹配。
3. eventKey 匹配。
4. 最近发现时间。

不引入向量检索。数据库规模很小，SQLite 普通查询和应用层轻量排序即可。

## 10. API 与 Service 建议

Route Handler 只做参数校验和响应转换，业务逻辑放在 Service。

建议增加或扩展：

```text
GET  /api/events/search?q=&excludeEventId=
GET  /api/events/[id]                         获取成员和聚类审计概况
POST /api/events/[id]/confirm                 确认当前事件归属
POST /api/events/[id]/move                    移动单篇 Article 到目标 Event
POST /api/events/[id]/split                   独立为新 Event
POST /api/events/merge                        合并整个 Event
PATCH /api/events/[id]                        指定代表 Article
POST /api/articles/[id]/recluster             单篇重新聚类
POST /api/articles/[id]/create-event          手动创建新 Event
```

实际实现可以在不造成职责混乱的前提下合并部分入口，但禁止让前端自行组合多次 API 来完成一次移动操作。

所有事件写操作继续使用现有 mutation guard，避免与批量 Job 并发修改 SQLite 事实。

## 11. 数据与审计要求

### 11.1 数据重建原则

- 不迁移历史 Article 与 Event 归属。
- 不回填历史 EventClusterAudit。
- 不转换历史 PushLog。
- 不保留旧聚类状态兼容逻辑。
- 如果数据结构需要调整，直接提交新的 Prisma migration。
- 部署改造后清空旧业务数据，重新抓取并生成新的 Article、Event 和审计记录。
- Setting、Source 等配置数据是否保留，应按实际结构单独判断，不得与 Article/Event 业务数据一起误删。
- 清理数据前仍需按部署规范备份数据库，防止误清配置数据。

### 11.2 EventClusterAudit 动作扩展

建议允许：

- `confirm`
- `move`
- `manual_create`
- `recluster`

已有动作继续保留：

- `create`
- `attach`
- `fallback_create`
- `merge`
- `split`
- `representative_change`

### 11.3 审计证据

人工操作 evidence 至少记录：

- 原 Event ID。
- 目标 Event ID。
- 操作前 clusterStatus。
- 原系统 decisionSource、confidence 和 evidence 摘要。
- 操作时间。

不需要新增复杂的人工反馈表。EventClusterAudit 足以承担当前规模的校准样本职责。

## 12. 状态与业务约束

必须保证：

- Article 最多属于一个 Event。
- 代表 Article 必须属于当前 Event。
- merged Event 不能作为移动、合并搜索候选。
- 移动、拆分、合并和删除后，Event.articleCount 必须与真实成员数一致。
- 空 Event 应按现有规则清理或进入明确终态。
- 人工确认后不得继续显示 `needs_review`。
- 聚类失败 Article 不能进入 AI、公开或推送。
- Event 已推送事实不能因成员或代表变化被清空。
- 新拆出的独立 Event 默认未推送。
- 修改代表文章不能改变历史 PushLog 展示内容。
- Event URL 使用 Event ID，代表文章变化不得改变公开 URL。

## 13. 前端交互约束

- 所有事件操作必须有明确 loading 状态，防止重复提交。
- 保存成功后再更新界面，不做不可回滚的乐观更新。
- 操作失败不得清空当前选择或关闭校准区域。
- 移动或拆分当前 Article 后，继续选中该 Article，并展示其新 Event。
- 合并整个当前 Event 后，如果当前 Article 被移入目标 Event，继续选中当前 Article。
- 事件搜索结果必须防止选择当前 Event。
- 危险操作确认框必须使用文章标题和事件标题，不以 ID 作为主要信息。
- 正常聚类依据默认折叠，异常依据默认展开。

## 14. 实施顺序

### 阶段一：可见性

- 左侧列表增加事件列。
- 增加聚类相关筛选。
- 右侧展示 Event 概况。
- 扩展 Event 详情接口返回最近聚类审计。
- 展示可读聚类依据。

完成标准：管理员能够理解 Article 当前聚类结果和形成原因。

### 阶段二：基础纠错

- 增加确认当前事件。
- 增加独立为新事件。
- 完善指定代表文章比较信息。
- 聚类失败增加重新聚类和手动创建事件。

完成标准：管理员可以处理待复核和聚类失败 Article。

### 阶段三：事件搜索与移动

- 实现 Event 搜索接口和轻量选择器。
- 实现移动 Article 到目标 Event。
- 合并整个 Event 改为搜索选择目标。
- 删除手工输入 Event ID 的主要交互。

完成标准：管理员不需要理解 Event ID 即可完成主要纠错。

### 阶段四：页面联动与收尾

- 抓取记录页增加“去校准”深链。
- 同事件成员改为比较表格。
- 完善操作后的局部刷新和选择保持。
- 更新 README、AGENTS 和相关测试。
- 清空旧 Article、Event、PushLog 和 EventClusterAudit 业务数据，重新采集首批文章。
- 首轮重新采集期间暂停实时推送，人工抽查聚类结果后再恢复。

完成标准：从发现异常到完成纠错形成完整闭环。

## 15. 验收标准

### 15.1 可见性

- 左侧列表无需打开详情即可识别待聚类、失败、待复核、多来源和代表文章。
- 右侧可以看到 eventKey、判断来源、置信度和可读证据。
- `needs_review` 与技术失败不会混淆。

### 15.2 人工操作

- 待复核 Article 可以确认属于当前 Event。
- Article 可以通过标题搜索移动到其他 Event。
- Article 可以独立为新 Event。
- 整个 Event 可以通过搜索选择目标后合并。
- 当前 Event 成员可以人工指定为代表 Article。
- 聚类失败 Article 可以重新聚类、移入已有 Event 或手动创建 Event。

### 15.3 一致性

- 操作后新旧 Event 的成员数、时间和代表文章正确。
- 公开列表和 Event URL 正确。
- 已推送状态不会被意外清空。
- 拆出的新 Event 能按资格正常首发。
- 历史 PushLog 不随当前代表文章漂移。
- EventClusterAudit 能完整追溯人工操作。

### 15.4 页面联动

- 抓取记录页的异常 Article 可以一键定位到情报收件箱。
- 情报收件箱自动选中目标 Article 并展开聚类区域。
- 操作后当前 Article 不丢失，页面无陈旧 Event 信息。

## 16. 测试重点

至少覆盖：

- 聚类状态和筛选投影。
- Event 搜索排除 merged 和当前 Event。
- 确认 `needs_review`。
- 单篇移动后新旧 Event 重算。
- 移动原代表 Article 后重新选择代表。
- 移动最后一篇 Article 后空 Event 处理。
- 已推送 Event 之间移动或合并不产生自动重复推送。
- 拆分的新 Event 保持未推送。
- 手动代表不被自动重算覆盖。
- 聚类失败重新聚类成功和再次失败。
- 页面深链、选择保持和失败不跳转。

## 17. 本轮不做

- 独立 Event 管理页面。
- 拖拽式跨 Event 移动。
- Embedding、向量数据库或语义搜索服务。
- EventRelation 和事件图谱。
- 自动训练或在线学习。
- 自动应用人工反馈修改提示词和规则。
- 聚类阈值可视化配置面板。
- 大规模 UI 重做。
- 历史 Article、Event、PushLog 和 EventClusterAudit 数据迁移或兼容。

## 18. 最终用户工作流

```text
抓取记录发现聚类失败或待复核
                ↓
点击“去校准”
                ↓
情报收件箱自动打开对应 Article
                ↓
查看当前 Event、成员和聚类依据
                ↓
确认当前事件 / 移到其他事件 / 独立为新事件
                ↓
必要时指定代表文章或合并整个事件
                ↓
系统重算 Event、公开状态并写入审计
```

最终效果应是：管理员通过文章标题、内容和来源完成事件纠错，不需要理解数据库关系，也不需要手工复制 Event ID。
