# Hot2-S 后台工作流与聚类校准剩余改造需求

> 文档用途：只记录当前代码尚未完成或尚未形成闭环的需求，供后续 AI 或开发人员直接实施。已完成能力不在本文重复展开；架构事实以 `README.md` 和 `AGENTS.md` 为准。

> 数据原则：按全新数据库和重新采集处理，不迁移、不回填、不兼容旧 Article、Event、PushLog 或 EventClusterAudit 数据。

## 1. 当前结论

后台职责拆分、Event 人工校准、单篇 workflow 入口、聚类审计基础结构、导航待办统计和面板深链主体已经存在，但以下边界仍未完整闭环：

1. 完整重推出现部分失败后，单篇 `push/retry` 可能被历史 `Event.pushedAt` 和历史成功日志短路，无法真正重试最新失败目标。
2. AI 依次判断多个聚类候选时，若前一个候选被拒绝、后一个候选被接受，最终 `attach` 审计会丢失前面的拒绝判断。
3. 导航技术待办按代表 Article 唯一计数，但抓取记录会把 Event 推送状态投影到所有成员 Article，失败数量和筛选口径不一致。
4. 收件箱已支持 `panel=content`，但抓取记录没有对应的跨页入口，URL 删除和浏览器导航状态也需要进一步收紧。
5. 抓取记录行虽然已删除评分展示，服务端快照和筛选仍保留软文、评分构成等业务判断字段。
6. 上述边界缺少专项回归测试。

剩余工作按 `P0 → P1 → P2` 实施。

## 2. P0：修复推送失败目标重试闭环

### 2.1 当前问题

单篇技术重试入口已经统一为：

```text
POST /api/articles/[id]/workflow
{
  "startAt": "push",
  "intent": "retry"
}
```

当前校验使用 `getFailedPushTargets(eventId)` 判断启用目标的最新投递是否失败，但执行阶段调用普通非强制推送：

```ts
pushArticleToFeishu(articleId, false)
```

`src/lib/push/delivery.ts` 仍存在两处冲突：

- `Event.pushedAt` 非空时，普通推送直接返回“已推送过”。
- 发送目标排除逻辑使用“历史上是否曾成功”，而不是“该目标最新一次投递是否成功”。

因此以下场景无法恢复：

```text
事件曾经全部推送成功
  ↓
管理员执行完整重新推送
  ↓
部分目标最新一次失败
  ↓
抓取记录显示失败并允许 retry
  ↓
实际执行被旧 pushedAt 或更早成功日志短路
```

### 2.2 目标方案

将推送意图从模糊的 `force: boolean` 收敛为明确模式。命名可结合现有代码调整，但必须保留以下语义：

```ts
type PushDeliveryMode =
  | 'normal'       // 流水线首次普通推送
  | 'retry_failed' // 只重试启用目标中最新状态失败的目标
  | 'repush_all'   // 管理员明确要求向全部启用目标完整重推
```

要求：

- `/api/articles/[id]/workflow` 的 `push + retry` 必须使用 `retry_failed`。
- Event 人工推送中的普通推送使用 `normal`。
- Event“完整重新推送”使用 `repush_all`。
- 不再让调用方通过布尔值猜测业务语义。

### 2.3 统一目标状态解析

在推送服务中建立唯一的“启用目标最新状态”解析函数，供以下位置共同使用：

- `validateSingleArticleWorkflow()`。
- `getFailedPushTargets()`。
- 实际推送目标选择。
- `getWorkQueueSummary()`。
- 抓取记录推送失败投影。

每个启用 Webhook URL 只读取最新一条 PushLog，形成：

```ts
{
  webhookUrl: string
  webhookRemark: string
  latestStatus: 'success' | 'failure' | 'never_attempted'
  latestCreatedAt: Date | null
}
```

不得再使用“存在任意历史成功日志”代替最新状态。

### 2.4 各模式行为

#### `normal`

- 遵守 Event 状态、代表 Article、聚类状态、AI 状态、评分和相关度门禁。
- 已经完整推送且当前无失败目标时不得重复发送。
- 只向尚未成功的启用目标发送。

#### `retry_failed`

- 只向最新状态为 `failure` 的启用目标发送。
- 不重复发送最新状态为 `success` 的目标。
- `Event.pushedAt` 是历史完整成功事实，不得阻止失败目标恢复。
- 仍遵守 Event、聚类、AI、评分和相关度门禁。
- 没有失败目标时返回 `409`，不得创建 Job。
- 若设置了重试时间，服务端和 UI 必须统一表达“等待重试”或“现在可重试”，不能创建必然被时间门禁拒绝的空 Job。

#### `repush_all`

- 只允许 Event 级人工操作触发。
- 明确向全部当前启用目标重新发送，不受历史成功日志影响。
- 不得通过 Article workflow 触发。
- 完整重推部分失败后，失败目标必须能够继续通过 `retry_failed` 恢复。

### 2.5 状态与结果记录

- 不为了重试清空历史 `Event.pushedAt` 或旧 PushLog。
- 最新目标状态以 PushLog 最新记录为准。
- 全部当前启用目标成功后，更新 `Event.pushedAt` 并清理 `nextPushRetryAt`。
- 部分或全部失败时保留成功目标事实，并更新 `nextPushRetryAt`。
- Job result 至少记录模式、尝试目标数、成功数、失败数和跳过数。

### 2.6 完成标准

- 完整重推部分失败后，单篇 `push/retry` 能实际重试最新失败目标。
- 已成功目标不会在失败恢复中收到重复消息。
- 历史成功、最新失败的目标不会被错误视为成功。
- 校验、导航统计、页面状态和实际投递使用同一目标状态解析结果。

## 3. P0：保存一次聚类决策中的全部 AI 候选判断

### 3.1 当前问题

`src/lib/event-clustering-service.ts` 已收集 AI 判断过的候选：

```ts
const aiCandidates = []
```

所有候选都被拒绝时，`fallback_create` 会保存完整 `aiCandidates`；但后续候选被接受时，`attach` 只保存最终接受候选的单个 `auditDecision`。

示例：

```text
候选 A：AI 判断不同事件
候选 B：AI 判断同一事件并接受
最终审计：只剩候选 B
```

这不满足“每次实际 AI 判断都可追溯”的要求。

### 3.2 目标审计结构

一次聚类执行只产生一个最终决策审计，但审计必须包含本次实际判断过的全部候选：

```ts
{
  selectedCandidateEventId: string | null
  candidates: Array<{
    candidateEventId: string
    ruleEvidence: {
      fingerprint: boolean
      exactTitle: boolean
      eventKeyMatch: boolean
      titleOverlap: number
    }
    aiDecision: {
      sameEvent: boolean
      confidence: number
      reason: string
      eventKey: string
    }
  }>
}
```

### 3.3 写入规则

- AI 接受第一个候选：`attach` 保存该候选判断。
- AI 拒绝前序候选、接受后续候选：`attach` 保存全部已判断候选，并用 `selectedCandidateEventId` 标记最终接受项。
- AI 拒绝全部候选：`fallback_create` 保存全部候选，`selectedCandidateEventId=null`。
- 未调用 AI 的 exact/rule 路径继续只保存规则证据，不伪造 AI 数据。
- 顶层 `candidateEventId` 在 `attach` 时指向最终接受 Event；`fallback_create` 可指向首要推荐候选，但完整事实以 `evidence.candidates` 为准。
- AI 拒绝候选时返回的 eventKey 只作为判断证据；不得因为拒绝判断而覆盖 Article 的最终 eventKey。

### 3.4 收件箱展示

收件箱聚类审计按判断顺序显示全部候选：

```text
候选 A：不同事件 · 58%
理由：财报周期不同

候选 B：同一事件 · 82% · 已采用
理由：主体、动作和事项一致
```

要求：

- 明确标记最终采用候选。
- 不把原始 JSON 作为唯一展示。
- 候选 Event 已删除或合并时仍保留 ID 和历史 AI 理由，不得导致详情接口报错。

### 3.5 完成标准

- 一次聚类中每个实际调用 AI 的候选都能从最终审计追溯。
- `attach` 和 `fallback_create` 使用同一 evidence 结构。
- 被拒绝候选的 eventKey 不会污染 Article 最终 eventKey。

## 4. P1：统一导航待办与抓取记录技术队列

### 4.1 当前问题

`getWorkQueueSummary()` 将推送失败映射到 Event 当前代表 Article，因此导航按唯一 Article 计数。

抓取记录则把 Event 的 `pushedAt` 和 `nextPushRetryAt` 投影到 Event 下所有 Article，造成：

- 同一个 Event 推送失败可能在页面被统计为多篇 Article 失败。
- 导航只计代表 Article，页面“含失败”可能计全部成员。
- `nextPushRetryAt` 到期后，页面状态从 `failed` 变为 `pending`，但最新 PushLog 仍然是失败，导航仍计技术待办。
- 导航数字和抓取记录筛选无法互相解释。

### 4.2 唯一技术队列口径

抽取服务端共享查询或共享解析器，返回唯一技术待办 Article 及原因：

```ts
type TechnicalIssue =
  | 'process_failed'
  | 'cluster_failed'
  | 'ai_failed'
  | 'push_failed'

{
  articleId: string
  issues: TechnicalIssue[]
  retryAvailableAt?: string | null
}
```

规则：

- process、cluster、AI 失败映射到发生失败的 Article。
- 推送失败只映射到 Event 当前代表 Article，不扩散到其他成员。
- 同一 Article 命中多个原因时，技术总数只计一次。
- 内容不足等业务终态的 AI skipped 不进入技术队列；只有明确的连续 AI 失败可进入。

### 4.3 抓取记录行为

- 增加或复用“技术异常”筛选，必须使用与导航相同的唯一 Article 集合。
- `含失败` 的计数不得把同一 Event 的成员重复计为推送失败。
- 代表 Article 的推送最新状态失败时始终显示失败事实：
  - 重试时间未到：显示“等待重试”。
  - 重试时间已到：显示“可重试”。
- 不得因为重试时间到期就把最新失败事实改成普通 `pending`。
- 非代表 Article 不提供 Event 推送重试操作；可显示“随事件处理”或 `not_applicable`。
- 导航全局总数与页面当前快照数量范围不同时，页面应明确标识“全局待办”和“当前快照”，不能声称两者天然相等。

### 4.4 人工队列保持现有口径

人工待办继续按唯一 Article 计数：

- `clusterStatus=needs_review`
- `reviewStatus=unreviewed`
- `aiStatus=done && aiConfidence < 70`

`manualOverrides != []` 不单独产生待办。

### 4.5 完成标准

- 同一 Event 的一次推送失败只产生一个代表 Article 技术待办。
- 导航技术数字与抓取记录“技术异常”筛选使用同一事实源。
- 推送失败不会因重试时间到期从技术队列中消失。
- 页面只在真正可恢复的 Article 上显示恢复操作。

## 5. P1：补全 `panel=content` 跨页入口与 URL 状态

### 5.1 当前问题

收件箱已经能读取并定位：

```text
/admin?tab=articles&articleId=<id>&panel=cluster
/admin?tab=articles&articleId=<id>&panel=content
```

但当前代码中只有抓取记录到 `panel=cluster` 的入口，没有生成 `panel=content` 的跨页链接。

同时，URL 同步仍需处理参数被删除的情况，避免浏览器前进、后退后 URL 已无 `articleId`，界面仍保留旧选中项。

### 5.2 目标交互

抓取记录保留一个技术详情入口，并补充清晰的业务查看入口：

- “去聚类复核” → `panel=cluster`
- “去内容校准”或“查看内容” → `panel=content`

跨页查看属于导航操作，不计入 Article 的“唯一下一步恢复操作”，不得与技术 retry 混为同一按钮。

### 5.3 `panel=content` 行为

- 精确加载目标 Article，即使不在当前分页。
- 滚动到 AI 洞察与内容校准区域。
- 若入口文案为“查看内容”，应自动展开正文核验区。
- Article 加载完成后再执行展开和滚动。
- 后续保存、归类或 Event 操作完成后保持当前 Article。

### 5.4 URL 状态规则

- URL 中存在 `articleId` 时，收件箱选中该 Article。
- URL 中删除 `articleId` 时，必须清理对应深链选中状态，不能只处理非空值。
- `panel` 被删除或值无效时，清理 `requestedPanel`。
- 页面内切换 Article 时同步更新 `articleId`，并按当前交互决定是否保留 `panel`。
- 切换后台 Tab 时清理另一页面专属参数。
- `AdminShell` 和收件箱共同响应应用内 URL 变化及 `popstate`，不得出现 Tab、URL 和详情状态不一致。

### 5.5 完成标准

- 抓取记录可以分别进入目标 Article 的聚类区和内容区。
- `panel=content` 能定位并按入口语义展开正文或内容校准区。
- URL 参数增加、变化、删除以及浏览器前进后退都能正确同步界面。

## 6. P2：彻底移除抓取记录中的内容质量判断

### 6.1 当前问题

抓取记录 Article 行已不展示评分构成和软文标记，但仍存在：

- `软文` 筛选项。
- 快照查询中的 `score`、`eventScore`、`contentScore`、`rawScore`、`adProbability`、`aiConfidence`、`category`、`isAd` 等业务字段。
- ArticleProgress DTO 中对应的评分与广告判断字段。

这些字段不属于 Job 监控或技术恢复，增加了两个页面的职责重叠。

### 6.2 目标方案

抓取记录仅保留技术监控所需字段：

- Article ID、标题、来源、发布时间和更新时间。
- fetch、cluster、AI、push 的技术状态与错误/重试时间。
- 当前唯一恢复操作所需的状态。
- 用于解释 AI skipped 的最小原因字段。
- Event 代表 Article 与最新推送失败事实所需的最小字段。

删除：

- `is-ad` / “软文”筛选。
- 行和 DTO 中不再使用的评分构成、广告概率、AI 置信度和内容分类字段。
- 只服务于内容质量判断的前端分支。

如果 `score` 和 `relevance` 仍是推送状态门禁计算必需字段，可以保留在服务内部投影输入中，但不得继续作为抓取记录 DTO 或交互能力暴露。

### 6.3 完成标准

- 抓取记录只回答“流程是否正常、失败在哪里、何时和如何恢复”。
- 评分、软文、内容分类和人工校准统一在情报收件箱查看。
- 服务端快照不再传输抓取记录未使用的业务字段。

## 7. P2：补充专项回归测试

### 7.1 推送恢复

- Event 历史已成功、完整重推最新部分失败时，`retry_failed` 仍会发送失败目标。
- 最新成功目标不会在 `retry_failed` 中重复发送。
- 历史成功但最新失败的目标按失败处理。
- 没有失败目标时 workflow 返回 `409` 且不创建 Job。
- `repush_all` 向全部启用目标发送，且只允许 Event 级入口触发。

### 7.2 聚类审计

- 第一个候选接受时保存完整 decision。
- 前序候选拒绝、后续候选接受时，`attach` 保存全部候选判断。
- 所有候选拒绝时，`fallback_create` 保存全部候选判断。
- 拒绝候选的 eventKey 不覆盖 Article 最终 eventKey。
- Event 详情能够为全部候选补充可读标题。

### 7.3 工作队列

- 同一 Event 多篇 Article 只由代表 Article 承担推送失败待办。
- 同一 Article 命中多个技术原因时总数只计一次。
- 推送失败在重试时间到期后仍属于技术待办。
- 导航与抓取记录技术筛选使用相同 Article ID 集合。
- `manualOverrides` 不单独产生人工待办。

### 7.4 深链与页面职责

- `articleId` 可以加载当前分页外的 Article。
- `panel=cluster` 和 `panel=content` 分别定位正确区域。
- `panel=content` 按入口语义展开正文核验区。
- URL 删除 `articleId` 或 `panel` 时清理相应状态。
- 删除“软文”筛选后，抓取记录筛选与快照 DTO 不再依赖内容质量字段。

## 8. 实施顺序

```text

P0-2 一次聚类执行的全部 AI 候选判断入审计

        ↓
P1-2 panel=content 入口与 URL 删除同步

        ↓
P2-2 补充专项回归测试并同步文档
```

## 10. 最终验收

全部满足后，本轮改造才可视为完整完成：

- 推送普通发送、失败目标重试和完整重推具有明确且不同的服务端语义。
- 历史成功不会掩盖最新投递失败，失败恢复不会重复发送成功目标。
- 聚类 AI 实际判断过的每个候选都能从最终审计追溯。
- 导航与抓取记录使用同一唯一技术待办事实源。
- 抓取记录能分别深链到聚类复核和内容校准区域。
- 抓取记录不再承担软文、评分和内容质量判断。
- 上述边界均有专项回归测试保护。
